## 输出要求
请拆解为多个子任务，每个子任务包含：
- task_id: 唯一标识（如 t1, t2, t3）
- task_type: 任务类型，必须严格使用以下枚举值之一：
  - rag_retrieval（知识库/文档检索）
  - db_query（数据库查询）
  - api_call（外部 API 调用，包括网络搜索、第三方服务等）
  - sandbox_coding（代码生成与脚本执行）
  - data_analysis（数据分析与可视化）
- risk_level: 风险等级（safe = 可信Worker执行 / dangerous = 沙箱执行）
- description: 任务描述
- input_data: 任务输入参数
- depends_on: 依赖的前置任务 ID 列表

## 规划原则
1. **最小任务原则**：能用 1 个任务完成的，绝不拆成多个。代码编写 + 测试属于同一个 sandbox 任务，不要拆开
2. **不要为代码任务添加 RAG 前置**：写代码不需要先检索"最佳实践"，LLM 自身就有这些知识
3. 数据获取类任务（检索、查询）优先执行，无依赖
4. 分析和代码生成任务依赖数据获取结果
5. 可视化和报告生成任务放在最后
6. 能并行的任务尽量并行（无依赖关系）
7. 代码生成、脚本执行必须标记为 dangerous

## 反面示例（不要这样做）
用户请求："用 Python 写快排并测试性能"
错误规划：t1 RAG检索快排 → t2 RAG检索测试方法 → t3 sandbox写代码 → t4 sandbox测试 → t5 分析
正确规划：t1 sandbox_coding（编写快排算法并进行性能测试，输出结果）— 只需要 1 个任务
