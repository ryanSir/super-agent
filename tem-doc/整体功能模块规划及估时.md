# 模块 1：API 网关（gateway/）

## 文件规划

| 文件 | 职责 |
|------|------|
| rest_api.py | REST 端点 + SSE 流 + 请求校验 |

## 端点清单

| 方法 | 路径 | 功能 |
|------|------|------|
| POST | /api/agent/query | 提交查询，异步编排 |
| POST | /api/agent/{session_id}/cancel | 取消正在执行的任务 |
| GET | /api/agent/stream/{session_id} | SSE 事件流（支持断点续传） |
| GET | /api/agent/skills | 列出已注册 Skill |
| POST | /api/agent/skills | 创建新 Skill |
| POST | /api/agent/admin/reload-mcp | 手动刷新 MCP 工具 |

## 核心功能点

1. **Worker 懒加载注册表** — importlib 动态加载 4 个 Worker（sandbox / db_query / api_call / web_search），失败跳过不阻塞启动
2. **全局状态管理** — configure() 由 lifespan 调用，初始化 ReasoningEngine / SessionManager / StreamAdapter
3. **查询提交** — 生成 session_id + trace_id，创建会话，asyncio.create_task 异步编排
4. **编排执行主流程** — 推理决策 → Agent 运行 → 结果提取 → 事件推送 → 状态更新
5. **Agent 运行：**
   - 按模式决定是否创建 Sub-Agent
   - 模型设置按通道分支：Claude 走 AnthropicModelSettings（extended thinking + budget_tokens），OpenAI 走 OpenAIModelSettings（max_tokens）
   - UsageLimits 限制（request_limit=15）
   - 超限降级：从消息历史提取最后文本
   - thinking 标签清理
6. **SSE 流端点** — 支持 Last-Event-ID 断点续传
7. **任务取消** — REST 接口取消正在执行的编排任务
8. **请求中间件** — auth / rate limit

## 技术栈

| 组件 | 选型 | 来源 |
|------|------|------|
| Web 框架 | FastAPI APIRouter | fastapi |
| Agent 框架 | pydantic-ai（iter + End 节点） | pydantic-ai |
| 异步执行 | asyncio.create_task | Python 标准库 |
| SSE 传输 | Starlette StreamingResponse | starlette（fastapi 内置） |
| SSE 生成器 | sse_event_generator（Redis Stream → SSE 格式） | 自研 |
| Anthropic 模型设置 | AnthropicModelSettings + extended thinking | anthropic SDK |
| OpenAI 模型设置 | OpenAIModelSettings | openai SDK |
| 模型路由 | llm/config.py 统一工厂，按模型名自动选通道 | 自研 |

# 模块 2：事件流（streaming/）

## 文件规划

| 文件 | 职责 |
|------|------|
| stream_adapter.py | Redis Streams 适配器，封装 XADD / XRANGE / XREAD |
| sse_endpoint.py | SSE 事件生成器，Redis Stream → SSE 格式推送 |
| protocol.py | 统一事件类型枚举 + 数据格式定义 |
| recovery.py | 断点续传 + 断连恢复机制 |

## 核心功能点

1. **StreamAdapter（Redis Streams 封装）：**
   - publish() — XADD 写入事件，自动注入 session_id / timestamp，maxlen 限流 + TTL 过期
   - read() — XRANGE 拉取历史事件，跳过 last_id 本身，支持断点续传
   - read_blocking() — XREAD 阻塞等待新事件，用于实时消费场景
   - Stream key 格式：`stream:{session_id}`
2. **SSE 事件生成器：**
   - 轮询 Redis Stream，批量读取（count=50）
   - 输出标准 SSE 格式：`id:` / `event:` / `data:`
   - 15 秒心跳保活，防止连接被中间件/代理断开
   - 遇到 session_completed / session_failed 自动结束流
   - 无事件时 0.5 秒间隔轮询
3. **事件协议：**
   - EventType 枚举（session_created / process_update / text_stream / tool_call / thinking / session_completed / session_failed）
   - 统一序列化/反序列化
   - 版本兼容
4. **断点续传：**
   - 客户端通过 Last-Event-ID 请求头传入断点
   - 服务端从该 ID 之后开始推送，不丢事件
   - 客户端重连后自动补发

## 技术栈

| 组件 | 选型 | 来源 |
|------|------|------|
| 消息队列 | Redis Streams（XADD / XRANGE / XREAD） | redis-py（aioredis） |
| SSE 格式输出 | AsyncGenerator → Starlette StreamingResponse | 自研 + starlette |
| 事件序列化 | json.dumps（ensure_ascii=False） | Python 标准库 |
| 心跳机制 | time.monotonic 计时 + 15s 间隔 | Python 标准库 |

## 估时

| 工作项 | 人天 |
|--------|------|
| StreamAdapter（publish / read / read_blocking + Redis 封装） | 1.5d |
| SSE 事件生成器（轮询 + 心跳 + 终止条件） | 1d |
| 事件协议定义（EventType 枚举 + 序列化） | 0.5d |
| 断点续传 + 断连恢复 | 1d |
| 单元测试 | 1d |
| **小计** | **5d** |

# 模块 3：会话管理（state/）

## 文件规划

| 文件 | 职责 |
|------|------|
| session_manager.py | 会话 CRUD + 状态机 + 对话历史持久化 |

## 核心功能点

1. **Session 数据模型：**
   - 字段：session_id / trace_id / query / status / created_at / metadata
   - `__slots__` 优化内存
   - to_dict() / from_dict() 序列化，metadata 用 JSON 字符串存 Redis Hash
2. **状态机：**
   - 状态流转：CREATED → PLANNING → EXECUTING → COMPLETED / FAILED / TIMEOUT
   - 合法转换表校验，非法转换打 warning 拒绝
3. **双层存储：**
   - 内存缓存（_local dict）+ Redis Hash 持久化
   - 读取优先查内存，miss 再查 Redis 并回填
   - TTL 1 小时自动过期
4. **会话 CRUD：**
   - create() — 创建会话，写内存 + Redis
   - get() — 双层查询
   - update_status() — 状态机校验后更新
5. **对话历史持久化：**
   - save_messages() — pydantic-ai 消息列表序列化存 Redis（key: `conversation:{session_id}:messages`）
   - load_messages() — 反序列化加载
   - 裁剪策略：保留最近 max_turns 轮（每轮 = 1 request + 1 response）
   - TTL 由 settings.memory.conversation_ttl 控制

| 持久化 | Redis Hash（HSET / HGETALL） | redis-py（aioredis） |

| 内存缓存 | Python dict | Python 标准库 |
| 状态机 | 自研转换表 + 枚举校验 | 自研 |

# 模块 4：推理引擎（orchestrator/）

## 文件规划

| 文件 | 行数 | 职责 |
|------|------|------|
| reasoning_engine.py | ~730 | 复杂度评估 + 模式路由 + 资源获取 + MCP 管理 |
| agent_factory.py | ~183 | 主 Agent 创建（pydantic-deep / fallback） |
| hooks.py | ~152 | 生命周期钩子（循环检测 + 安全审计） |
| planning.py | ~39 | DAG 规划 Prompt 模板 |

## 核心功能点

1. **模式决策四级流水线（_resolve_mode）：**
   - Level 0：用户显式指定模式，直接返回
   - Level 1：高置信度快筛（≤30 字 + 无任务动词 → DIRECT）
   - Level 2：LLM 主分类（Classifier Agent，temperature=0，超时降级）
   - Level 3：规则兜底（正则模式匹配 + 五维度复杂度评估）
2. **五维度复杂度评估（LLM 失败时的降级路径）：**
   - task_count（0.25）— 动词计数估算子任务数
   - domain_span（0.20）— 跨领域关键词共现（6 个领域）
   - dependency_depth（0.20）— 顺序连接词层级
   - output_complexity（0.15）— 输出类型复杂度
   - reasoning_depth（0.20）— 推理类型深度
   - 加权分数 → 四级复杂度 → 四种执行模式
3. **资源获取与缓存（_resolve_resources）：**
   - 桥接工具（基于 Workers 创建，分组 dict）
   - MCP toolsets（pydantic-ai MCPServer 实例）
   - Skill 摘要（由框架自动管理）
   - 缓存复用，MCP 刷新后自动失效重建
4. **MCP 生命周期管理：**
   - startup 时创建 MCP Server + 预热资源缓存
   - 定期刷新任务（可配置 interval）
   - 手动刷新接口（reload_mcp）
   - shutdown 时取消刷新任务
5. **Agent 工厂（agent_factory.py）：**
   - 基于 pydantic-deep create_deep_agent() 创建主 Orchestrator
   - 动态 System Prompt 构建（12 段模板）
   - Hooks 注入（循环检测 + 审计）
   - Capabilities 注入（事件发布 + ToolGuard 黑名单）
   - DIRECT 用 fast 模型，复杂模式用 planning 模型
   - MCP toolsets + 桥接工具 + Sub-Agent 配置一并注入
   - Fallback：pydantic-deep 不可用时降级到原生 PydanticAI Agent
6. **生命周期钩子（hooks.py）：**
   - 循环检测：滑动窗口 + MD5 指纹去重，warn_threshold 警告 + hard_limit 强制拒绝
   - 安全审计：工具调用开始/完成/失败全链路日志（工具名 / 参数 / 耗时 / 结果）
   - 按配置开关注册
7. **DAG 规划 Prompt（planning.py）：**
   - 6 种任务类型（db_query / api_call / sandbox_coding / data_analysis / sub_agent_task）
   - 7 条规划原则（最小任务 / 无冗余前置 / 数据优先 / 最大并行 / 风险标记等）

## 技术栈

| 组件 | 选型 | 来源 |
|------|------|------|
| Agent 框架 | create_deep_agent() | pydantic-deep |
| Fallback Agent | pydantic-ai Agent | pydantic-ai |
| LLM 分类器 | pydantic-ai Agent + ModelSettings(temperature=0) | pydantic-ai |
| 模型路由 | get_model() 双通道（Anthropic / OpenAI） | 自研（llm/config.py） |
| 安全防护 | ToolGuard 工具黑名单 | pydantic-ai-shields |
| 钩子框架 | Hook / HookEvent / HookInput / HookResult | pydantic-deep |
| 正则匹配 | 中英文双语模式库 | Python 标准库 re |
| MCP 管理 | mcp_client_manager | 自研（capabilities/mcp/） |

## 估时

|--------|------|
| 模式决策四级流水线（快筛 + LLM 分类 + 规则 + 复杂度） | 3d |

# 模块 6：Sub-Agent（agents/）

## 文件规划

| 文件 | 职责 |
|------|------|
| roles.py | 3 个预置角色的 System Prompt（researcher / analyst / writer） |
| models.py | Sub-Agent 输入输出契约（SubAgentInput / SubAgentOutput） |

## 核心功能点

1. **3 个预置角色（roles.py）：**
   - researcher — 信息检索与综合分析（RAG / API / Skill / Sandbox）
   - analyst — 数据分析与可视化（SQL / RAG / Sandbox / ECharts）
   - writer — 报告与文档撰写（Skill / Sandbox / ECharts）
   - 每个角色有独立的职责、可用工具、工作方式、输出要求指令
2. **输入输出契约（models.py）：**
   - SubAgentInput — task_id / sub_agent_name / instruction / context / parent_results / timeout
   - SubAgentOutput — task_id / success / answer / data / artifacts / token_usage / error
   - 自包含，不依赖主 Agent 内部状态，方便未来升级 multi-agent
3. **配置工厂（factory.py）：**
   - 合并所有 base tools（排除 plan 组，sub-agent 不应自己规划 DAG）
   - 统一注入：全量工具 + MCP toolsets + SkillsToolset
   - 角色差异化通过 instructions 控制，不限制工具集
   - 每个 sub-agent 配置：model(subagent) / todo / skills / context_manager(100k) / cost_tracking
   - 预置 3 个 + 自定义合并输出
4. **自定义 Agent 注册表（custom/registry.py）：**
   - 目录扫描：遍历 agents/ 下的子目录，发现 AGENT.md
   - YAML frontmatter 解析：name / description / model / tools
   - frontmatter 之后的内容作为 instructions
   - 转换为 SubAgentConfig 兼容字典
   - 全局单例 custom_agent_registry

## 技术栈

|------|------|------|
| 数据模型 | Pydantic BaseModel | pydantic |
| Agent 配置 | create_deep_agent 兼容 dict | pydantic-deep |
| 模型路由 | get_model("subagent") | 自研（llm/config.py） |
| 配置解析 | YAML frontmatter 正则 + pathlib 目录扫描 | Python 标准库 |

## 核心功能点

1. **Base Tools 桥接层（base_tools.py）：**
   - 将 Workers 包装为 RunContext[Any] 签名的工具函数
   - 每个工具构造 TaskNode → 调用 Worker.execute() → 返回 model_dump()
   - 按职责分组返回 dict，Sub-Agent 共享同一批实例（排除 plan 组）

2. **事件发布 Capability（event_publishing.py）：**
   - 基于 pydantic-ai AbstractCapability 实现
   - wrap_tool_execute() — 发布 tool_call + tool_result + Langfuse span + 计时
   - on_tool_execute_error() — 发布 error 状态 + Langfuse ERROR span
   - after_model_request() — 从 LLM 响应提取 thinking + text_stream 事件
   - after_run() — 发布流终止标记（is_final=True）
   - emit_chart 特殊处理：额外推送 render_widget 事件
   - 工具类型自动推断（sandbox / native_worker / skill / builtin / mcp）
   - tool_call_id 去重，防止重复推送

3. **Skill 系统：**
   - SkillMetadata — name / description / path / execution（native | sandbox）
   - SkillInfo — metadata + scripts + references + doc_content
   - SkillRegistry — 目录扫描 + YAML frontmatter 解析（支持 yaml 库 + 正则 fallback）
   - 三阶段渐进加载已委托给 pydantic-deep SkillsToolset
   - register() 支持动态注册（create_skill 后立即生效）

## 核心功能点

### 1. Worker 基类（base.py）
- **WorkerProtocol** — runtime_checkable 协议，定义 name + execute() 接口

### 2. 4 个 Native Workers
- **RAGWorker** — Milvus 向量检索，懒初始化客户端，同步搜索包装为 run_in_executor 异步
- **DBQueryWorker** — SQL 只读查询，正则黑名单校验（DROP/DELETE/INSERT/UPDATE 等 11 个关键词），违规直接拒绝

- 销毁沙箱（finally 保证清理）

### 4. SandboxManager（sandbox_manager.py）— 双后端透明切换
- **Local 后端**：tempfile.mkdtemp 创建临时目录，asyncio.create_subprocess_shell 执行，实时逐行读取 stdout/stderr，超时 kill 进程
- **E2B 后端**：e2b.Sandbox 创建云沙箱，commands.run 执行
- **统一接口**：create() / execute() / write_file() / read_file() / destroy()
- **配置切换**：E2B_USE_LOCAL=true 走本地

### 5. Pi Agent 配置（pi_agent_config.py）
- 启动脚本模板：加载 nvm → 设置环境变量 → 写入 instruction 文件 → `script -q /dev/null pi --print --mode json` 执行
- 原子工具集：read,bash,edit,write（沙箱内仅提供最底层能力）
- 支持自定义 provider / model / tools
- 业务 key 注入（BAIDU_API_KEY 等）

### 6. JSONL IPC 解析（ipc.py）
- `parse_jsonl_output()` — 逐行解析 JSON，容错跳过
- `extract_final_answer()` — 三级 fallback：agent_end → turn_end/message_end → message_update(text_end)
- `extract_artifacts()` — 提取 artifact 事件 + tool_result 中的文件产物

- **RedisMemoryStorage 实现**：
  - Profile → Redis Hash（`memory:{user_id}:profile`）
  - Facts → Redis Sorted Set（`memory:{user_id}:facts`），score=timestamp，按时间倒序

### 4. 记忆更新（updater.py）
- 分布式锁：Redis SET NX EX，防止并发更新同一用户记忆
- LLM 并行抽取：`_extract_profile()` + `_extract_facts()` 用 asyncio.gather 并行
- Profile 抽取：LLM 从对话中提取 work_context / personal_context / top_of_mind
- Facts 抽取：LLM 从对话中提取最多 5 条事实
- 去重：与已有 facts 内容比对，跳过重复
- 上限控制：超过 max_facts 时驱逐最旧记录（预留）

| TaskType | 6 种任务类型（rag / db / api / sandbox / analysis / sub_agent） |
| RiskLevel | safe / dangerous，决定路由到可信 Worker 还是沙箱 |
| TaskStatus | pending / running / completed / failed / skipped |
| SessionStatus | created / planning / executing / completed / failed / timeout |
| EventType | 14 种 SSE 事件类型（含 Sub-Agent 生命周期） |

### DAG 模型

| 模型 | 职责 |
|------|------|
| TaskNode | 单个任务节点（id / type / risk / input / depends_on / status） |
| ExecutionDAG | 任务拓扑图（task_map 索引 + root_tasks 属性） |

### 通信模型

| 模型 | 职责 |
|------|------|
| AgentMessage | Agent 间通信消息（role / content / metadata） |

### A2UI 模型

| 模型 | 职责 |
|------|------|
| A2UIFrame | 事件帧基类（event_type / trace_id / session_id / timestamp） |
| TextStreamFrame | LLM 文本流（delta + is_final） |
| RenderWidgetFrame | 渲染组件指令（widget_id / ui_component / props） |
| ToolResultFrame | 工具结果（tool_name / tool_type / status / content） |

### 沙箱模型

| 模型 | 职责 |
|------|------|
| SandboxTask | 沙箱任务（instruction / context_files / timeout） |
| Artifact | 沙箱产物（name / path / content_type / size） |
| SandboxResult | 沙箱结果（stdout / stderr / exit_code / artifacts） |

## 核心功能点

1. **DAG 任务模型** — TaskNode 支持依赖链（depends_on），ExecutionDAG 提供 task_map 索引和 root_tasks 便捷属性
2. **A2UI 协议** — 统一信封格式，所有 SSE 事件继承 A2UIFrame 基类，`to_event_dict()` 自动处理 datetime 序列化
3. **QueryRequest 模式校验** — mode 字段用 regex pattern 约束合法值
4. **OrchestratorOutput** — 作为 PydanticAI result_type，聚合 answer / plan / worker_results / sub_agent_results / a2ui_frames
5. **沙箱模型自包含** — SandboxTask / SandboxResult / Artifact 独立于 Agent 核心模型，方便沙箱层独立演进

- AgentError 基类（message + detail）
- 7 个分支：
  - ReasoningError
  - SubAgentError（+Timeout）
  - WorkerError（+Routing）
  - BridgeError
  - SandboxError（+Timeout+Execution）
  - LLMError（+RateLimit+TokenExceeded）
  - PlanningError / MemoryError
