# Super-Agent 技术选型结论摘要

> 版本：v1.0
>
> 日期：2026-04-17

## 1. 结论

**结论：建议选择 `pydantic-deep v0.3.3` 作为 Super-Agent 的底层 Agent SDK。**

这不是因为它在所有开源框架里综合能力最强，而是因为在本轮评估范围内，它是**与目标架构最匹配、集成成本最低、可控性最高**的方案。

同时，建议采用以下组合策略：

- **底层编排内核**：`pydantic-deep`
- **安全与工程标准参考**：`DeerFlow`
- **多 Agent 与记忆系统参考**：`AgentScope`
- **执行抽象与权限模型参考**：`Deep Agents`

一句话概括：

**底座选 `pydantic-deep`，工程标准学 `DeerFlow`，多 Agent 能力参考 `AgentScope`，安全抽象参考 `Deep Agents`。**

---

## 2. 为什么选 `pydantic-deep`

### 原因 1：编排模型最匹配

Super-Agent 目标是“控制面负责路由和模式决策，执行面负责自主完成任务”。  
`pydantic-deep` 采用“LLM 自主循环 + `task()` 委派”，与这一架构分工天然一致。

相比之下：

- `CrewAI` 偏开发者显式编排，不适合动态路由
- `Deep Agents` 虽然能力强，但 LangGraph / LangChain 绑定更深

### 原因 2：依赖最轻，升级风险最低

- `pydantic-deep` 核心依赖较轻，主要围绕 `pydantic-ai`
- `Deep Agents` 依赖 LangChain 生态较深，升级波动更大
- `CrewAI` 虽稳定，但架构契合度不足

### 原因 3：可替换性最好

已有分析表明，该方案保留了降级到原生 `pydantic-ai` 的可能性。  
这意味着底层 SDK 不是不可替代的，技术路线更稳。

### 原因 4：源码规模可控

`pydantic-deep` 体量相对较小，团队可读、可控、可 fork，自维护成本可接受。

---

## 3. 为什么不直接选其他框架

### `DeerFlow`

优点：

- 工程成熟度最高
- 中间件体系最好
- 安全审计和错误处理最强

不作为底座的原因：

- 更像完整平台，不像轻量 SDK
- 对 LangChain / LangGraph 依赖较重
- 更适合作为工程设计标杆，而不是直接嵌入

### `AgentScope`

优点：

- 多 Agent 协作最成熟
- 记忆系统最完整
- 状态管理与观测能力很强

不作为底座的原因：

- 更偏平台型框架
- 无内置沙箱，安全侧有明显短板
- 核心模块较重，不适合作为本阶段编排内核

### `Deep Agents`

优点：

- 执行后端抽象最干净
- 权限模型明确
- 上下文压缩做得好
- 有正式威胁模型意识

不作为底座的原因：

- API 变化快
- pre-1.0 风险较高
- LangChain 依赖链较深

### `CrewAI`

优点：

- 商业化程度更高
- API 稳定性相对更好

不作为底座的原因：

- 开发者显式编排模式与目标架构不匹配
- 灵活性不足，场景切换成本高

---

## 4. 风险判断

### 可以接受的风险

- `pydantic-deep` 仍处于 pre-1.0 阶段
- API 稳定性不如成熟大版本产品
- 某些高级能力不如平台型框架完整

### 当前最需要关注的风险

不是“底层选错了”，而是“工程化补强还没做完”，主要包括：

- Beta API 依赖风险
- 静默吞异常风险
- 默认能力开启过多带来的误调用与性能损耗
- 少量私有属性访问带来的升级断裂风险

结论是：

**当前的主要问题不是换框架，而是先完成产线硬化。**

---

## 5. 上线前建议

建议先完成 4 个低成本动作：

1. 锁定关键依赖版本
2. 用稳定 API 替换 Beta API 配置
3. 关闭不必要的默认能力
4. 把静默异常改为可观测日志

这四项投入很小，但对稳定性提升明显。

---

## 6. 中长期建议

### 短期

继续以 `pydantic-deep` 为底层推进，优先完成产线硬化和主链路稳定。

### 中期

重点补齐：

- 安全审计
- 错误处理分级
- 监控与追踪
- 权限模型

这里应重点借鉴 `DeerFlow` 和 `Deep Agents`。

### 长期

若目标升级为企业级 Agent 平台，应重点演进：

- 多 Agent 协作
- 长期记忆
- 状态序列化
- 跨服务协同

这里应重点参考 `AgentScope`。

---

## 7. 最终建议

最终建议可以压缩为三句话：

1. **底层 Agent SDK 选择 `pydantic-deep`。**
2. **当前阶段的重点不是更换框架，而是完成产线硬化和工程补强。**
3. **长期竞争力不取决于底层 SDK 本身，而取决于能否把 `DeerFlow`、`AgentScope`、`Deep Agents` 的优点吸收到 Super-Agent 自身架构里。**
