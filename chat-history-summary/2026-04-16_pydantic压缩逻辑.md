# 对话摘要：pydantic-deep 自动压缩机制分析

> 保存时间：2026-04-16
> 原始轮次：4 轮

## 背景

项目通过 `create_deep_agent(context_manager=True, context_manager_max_tokens=200_000)` 启用了框架自动压缩，需要了解其内部实现逻辑、压缩提示词，以及是否支持自定义压缩策略。

## 核心内容

- 压缩能力由 `pydantic_ai_summarization` 包（v0.1.3）的 `ContextManagerCapability` 提供
- 触发链路：`before_model_request` → `count_tokens_approximately(字符数÷4)` → 达到 `compress_threshold`（默认 90%）时触发
- 压缩执行：`SummarizationProcessor` 确定 cutoff 点（避免切断 tool_call/tool_return 配对）→ 格式化为文本 → 用独立 agent（默认 gpt-4.1-mini）生成摘要 → 包装为 `SystemPromptPart` 替换旧消息
- 压缩提示词采用 XML 结构化格式，角色为 "Context Extraction Assistant"，指令要求只提取关键上下文、避免重复已完成动作
- 框架支持丰富的自定义参数，无需写自定义 Capability：`compress_threshold`、`summarization_model`、`summary_prompt`、`keep`、`token_counter`、`on_before_compress`/`on_after_compress` 回调、`include_compact_tool`（让 agent 主动触发压缩）
- 如需完全自定义压缩逻辑，可关闭 `context_manager=False`，自己实现 `AbstractCapability` 的 `before_model_request`

## 后续事项

- 考虑将 `summarization_model` 从默认的 `gpt-4.1-mini` 切换为项目自有模型路由（`get_model("subagent")`）
- 评估是否需要自定义 `summary_prompt` 以更好适配 Agent 编排场景（当前默认 prompt 偏通用）
- 评估是否启用 `include_compact_tool=True` 让 agent 在长对话中主动压缩
