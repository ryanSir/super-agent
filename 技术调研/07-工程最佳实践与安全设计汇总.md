# 工程最佳实践与安全设计汇总

本文用于持续整理 Agent 系统在工程落地中的核心设计原则、最佳实践与安全边界。

> 本文关注的是带有模型调用、工具执行、权限控制、沙箱隔离、流式交互的 Agent 系统，重点回答以下问题：
>
> - 主循环应该如何设计
> - 模型输出为什么不能直接视为授权
> - 权限审批和执行控制如何解耦
> - 为什么沙箱对网络出口控制是必要的
> - 哪些工程约束应该默认内建，而不是事后补丁
>
> 如果只保留几条最高优先级原则，建议先记住这些：
>
> - 模型输出不是授权，系统必须独立完成权限校验
> - 主循环必须显式建模成状态机，而不是若干隐式回调拼装
> - 工具调用必须经过风险识别、审批、执行、回写四个阶段
> - 沙箱负责约束实际运行行为，不能只依赖命令级规则
> - 默认拒绝、最小权限、可审计、可回放，应该是基础能力

___

## 1. 核心循环设计：`query.ts` 的 AsyncGenerator 状态机

如果只能从 CC 源码里学一件事，这一部分值得优先看：`query.ts` 里的 Agentic Loop 实现方式。

它不是简单的“调一次模型，拿一次结果”，而是把多轮推理、工具调用、状态推进、流式事件分发，统一进一个可追踪的状态机里。

### 1.1 为什么这样设计

Agent 系统最容易失控的地方，不是 Prompt，而是主循环。

一个不清晰的主循环通常会带来这些问题：

- 多轮调用之间状态被隐式污染
- 工具执行和模型响应耦合过深
- UI 流式更新依赖大量回调嵌套
- 压缩、终止、重试逻辑难以插入
- 调试时很难还原“第几轮发生了什么”

CC 在这里的做法比较干净：把 loop 明确建模成状态机，并用 `AsyncGenerator` 向上层持续产出事件。

这一节如果继续优化排版，更适合按三层来组织：

- 为什么这样设计
- 具体怎么实现
- 可以沉淀出什么最佳实践

这样读者先理解设计动机，再看实现细节，最后再抽象出自己的工程方法。

### 1.2 具体做法

#### 1.2.1 用不可变 `State` 管主循环

`query loop` 维护一个不可变的 `State` 对象。每一轮迭代不修改旧状态，而是基于当前状态构造新状态。

```ts
// query.ts — State 对象（概念结构，已简化）
interface State {
  messages: MessageParam[]        // 完整对话历史（不可变追加）
  turnCount: number               // 当前对话轮次
  shouldAutoCompact: boolean      // 是否触发自动压缩
  autoCompactTracking: {
    consecutiveFailures: number   // 连续压缩失败次数
    totalMessages: number         // 压缩前消息总数
  }
  aborted: boolean                // 是否已被 AbortController 中止
}

// 状态转换：纯函数，不可变
function next(current: State, reason: string, overrides?: Partial<State>): State {
  return { ...current, ...overrides }
}
```

这里最关键的设计选择是“不可变 State”。

原因很直接：

- Agentic Loop 是多轮迭代，状态跨度长
- 一旦允许就地修改，问题定位会迅速恶化
- 压缩、重试、中止都会让状态分支变复杂

不可变设计的价值在于：

- 每一轮状态转换都是显式的
- 更容易记录和回放
- 更适合调试、审计和测试
- 更容易在未来加入快照、checkpoint、恢复执行

#### 1.2.2 用 `AsyncGenerator` 推进 `queryLoop()`

`queryLoop()` 是一个 `async function*`，本质上是一个持续推进状态的异步生成器。

```ts
// query.ts:241 — 概念简化版
async function* queryLoop(state: State, tools: Tool[], systemPrompt: string) {
  while (true) {
    const stream = await callAPI(state.messages, tools, systemPrompt)

    for await (const chunk of stream) {
      yield { type: 'stream_chunk', data: chunk }
    }

    const toolUses = extractToolUses(stream.finalMessage)
    if (toolUses.length === 0) {
      return
    }

    const toolResults = await executeTools(toolUses, state)
    yield { type: 'tool_results', data: toolResults }

    state = next(state, 'tool_results', {
      messages: [...state.messages, stream.finalMessage, ...toolResults],
      turnCount: state.turnCount + 1
    })

    if (shouldCompact(state)) {
      state = await runCompaction(state)
    }
  }
}
```

这段实现把三件本来很容易散掉的事情，合成在一个统一控制面里：

- 多轮迭代
- 流式推送
- 状态转换

#### 1.2.3 用一条请求看清完整运行路径

假设用户输入：

`帮我分析一下这个项目的目录结构`

```text
用户输入 → REPL.tsx:onSubmit
    ↓
handlePromptSubmit.ts → processUserInput
    ↓
构建三路上下文（并行）：
  ├── getSystemPrompt()      → System Prompt（5 级优先级决策）
  ├── getSystemContext()     → gitStatus + Agent 工具列表
  └── getUserContext()       → CLAUDE.md + 当前日期
    ↓
query.ts:query() → queryLoop(state, tools, systemPrompt)
    ↓
[轮次 1] API 请求 → Claude 回复："我需要先看看目录结构"
       → 工具调用：Bash("ls -la")
       → 权限检查：Allow（只读命令）
       → 执行：返回目录列表
       → yield tool_result → UI 显示执行过程
    ↓
[轮次 2] API 请求（带 ls 结果）→ Claude 回复："目录结构如下..."
       → 无工具调用
       → yield 最终回复 → UI 显示结果
    ↓
queryLoop 退出（return）
```

UI 层只需要：

```ts
for await (const event of queryLoop(...)) {
  render(event)
}
```

这也是为什么它的流式体验会比较自然。上层不需要知道 loop 内部到底经历了几轮模型调用、几次工具执行、是否发生了压缩，只需要消费事件。

#### 1.2.4 为什么 `AsyncGenerator` 适合 Agentic Loop

很多 Agent 实现会把以下逻辑拆得到处都是：

- 模型流式 token 回调
- 工具调用回调
- 权限批准回调
- 最终响应回调

最后主循环被回调和事件监听撕碎，调试非常痛苦。

`AsyncGenerator` 的好处是，它天然适合“一个长生命周期流程，不断向外产生中间事件”的场景：

- 内部仍然可以保留顺序化控制流
- 外部可以按事件流逐步消费
- UI、日志、trace、测试都能复用同一事件接口
- 工具结果、模型分块、审批事件都能统一建模

### 1.3 可以沉淀出的最佳实践

- 主循环先建模成状态机
- `State` 尽量不可变
- 事件输出尽早统一化
- 工具执行视为正式状态转换阶段
- 给压缩、中止、重试预留明确挂点
- 为每轮迭代记录 trace，而不是只保留最终结果

## 2. 上下文管理：三路并行组装与 System Prompt 设计

CC 的上下文管理不是把所有信息粗暴拼成一个大 prompt，而是把“哪些内容放进 system”“哪些内容放进 user”“哪些片段应该缓存”“哪些片段必须动态重算”拆得非常清楚。

这部分的价值不只是 Prompt 工程，而是工程化的上下文编排能力。它决定了系统的稳定性、成本、缓存命中率，以及不同来源指令之间的优先级边界。

### 2.1 为什么这样设计

Agent 系统的上下文管理如果没有明确分层，通常会出现这些问题：

- 所有上下文都被拼进同一段文本，来源和优先级不清
- 明明是静态规则，却在每轮请求里重复计费
- 项目记忆、用户记忆、系统指令相互覆盖，难以解释最终行为
- 工具列表、git 状态、日期等动态信息污染缓存命中率
- 安全规则和业务规则混写在一起，维护时容易被误删

CC 的做法是把上下文拆成三路并行组装，并进一步在 `System Prompt` 内部切分静态段和动态段。这使它同时满足：

- 语义分层清晰
- token 成本可控
- prompt cache 命中率更高
- 指令优先级可解释
- 安全规则可独立维护

### 2.2 具体做法

#### 2.2.1 每轮请求并行构建三路上下文

每次用户提交消息，CC 会并行构建三路上下文，而不是串行逐段拼接：

```ts
// REPL.tsx:2768-2789 — 三路并行
const [, , defaultSystemPrompt, baseUserContext, systemContext] = await Promise.all([
  checkAndDisableBypassPermissionsIfNeeded(...),
  checkAndDisableAutoModeIfNeeded(...),
  getSystemPrompt(freshTools, ...),     // ① System Prompt
  getUserContext(),                     // ② User Content
  getSystemContext()                    // ③ System Content
])
```

三路分工很清晰：

- `System Prompt`：承载身份、规则、风格、任务执行原则等高优先级指令
- `System Content`：承载工具、git 状态、MCP server、Skills 等运行时系统信息
- `User Content`：承载 `CLAUDE.md`、日期、用户消息等偏用户侧上下文

这种拆法的关键，不只是并行提速，而是把“规则”“环境”“用户输入”从语义上拆开。

#### 2.2.2 `System Prompt` 使用明确的优先级覆盖链

`buildEffectiveSystemPrompt()` 不是简单地选一个 prompt，而是按优先级决定谁覆盖谁。

从高到低可以概括为：

- `overrideSystemPrompt`
- `Coordinator prompt`
- `Agent prompt`
- `customSystemPrompt`
- `defaultSystemPrompt`
- `appendSystemPrompt`

其中：

- `overrideSystemPrompt` 用于 loop 模式或特殊场景，直接完全替换
- `Coordinator prompt` 在协调器模式启用时生效
- `Agent prompt` 用于当前是自定义 Agent 的场景
- `customSystemPrompt` 由 `--system-prompt` 显式指定
- `defaultSystemPrompt` 是标准默认提示词
- `appendSystemPrompt` 始终附加在末尾

这套覆盖链的价值在于：系统知道“为什么现在生效的是这一版 prompt”，而不是靠若干隐式字符串拼接碰运气。

#### 2.2.3 `System Content` 只放运行时环境信息

`getSystemContext()` 注入到 system 消息末尾，主要负责提供当前运行环境的动态信息，例如：

- `gitStatus`
- Agent 工具可用列表，以及每个工具的 `whenToUse`
- MCP server 列表
- Skills 列表

这里尤其值得注意的是 `gitStatus`。它属于典型的动态信息，必须每轮重算，因此应明确标记为 `uncached section`，而不应混进可缓存的静态 prompt 里。

#### 2.2.4 `User Content` 负责记忆与用户态上下文

`getUserContext()` 会包装成第一条 user 消息，包含：

- `CLAUDE.md`
- 当前日期
- 用户消息

其中 `CLAUDE.md` 的加载顺序是分层的：

1. Managed memory
2. User memory
3. Project memory
4. Local memory

一个关键设计是：`CLAUDE.md` 被放进 user 消息，而不是 system 消息。

这样做不是语义偶然，而是缓存策略设计。这样可以让 `CLAUDE.md` 作为相对稳定的用户侧记忆独立命中 prompt cache，而不污染高优先级 system prompt 的缓存结构。

#### 2.2.5 在 `System Prompt` 内再切一刀：静态段和动态段分离

`getSystemPrompt()` 返回的是 `string[]`，而不是一个单一大字符串。每个片段都可以独立参与缓存。

结构上可以理解为：

```text
[静态段，全局可缓存]
  ├── 身份介绍
  ├── 系统行为规范
  ├── 任务执行指南
  ├── 操作安全指南
  ├── 工具使用指南
  ├── 语调风格
  └── 输出效率

SYSTEM_PROMPT_DYNAMIC_BOUNDARY

[动态段，每轮重算]
  ├── Agent 工具、Skills 上下文
  ├── memory
  └── gitStatus
```

这条静动态分界线非常关键。因为真正高频变化的是后半段，而前半段往往跨 session 都是稳定的。

这意味着：

- 静态规则可以跨 session 全局缓存
- 动态运行时信息单独重算
- 每次请求不必为整段 prompt 重复付费

在高频调用场景下，这不是小优化，而是直接影响成本结构的核心设计。

#### 2.2.6 `defaultSystemPrompt` 不只是提示词，更是工程规则载体

CC 的 `defaultSystemPrompt` 本身就很值得拆开看，其中有几个工程细节很典型。

第一，安全规则独立维护，并用 `IMPORTANT` 作为强标记。

例如：

- 授权安全测试边界
- 禁止生成或猜测 URL

这类规则来自独立文件，由专门团队维护。工程启示很明确：安全规则和业务规则不要混在一起维护，否则一次普通重构就可能误删安全边界。

第二，工具指南写的是“什么时候用什么工具”，而不只是“工具能做什么”。

这意味着 prompt 不只是能力清单，更是工具路由策略。它直接影响模型如何选择：

- 是直接用 `Glob/Grep`
- 还是调用更重的 Explore agent

第三，输出效率被单独抽成一节。

这说明“怎么回答才算高效”也是一等工程约束，而不是依赖模型自行揣摩。把它写在 prompt 层，而不是固化在模型层，意味着后续更容易按产品形态做定制和覆盖。

### 2.3 可以沉淀出的最佳实践

- 上下文按语义拆层，不要把所有信息堆进一个大 prompt
- `System Prompt`、运行时环境、用户侧记忆分开组装
- 让优先级覆盖链显式化，而不是依赖隐式字符串拼接
- 静态段和动态段明确分界，优先优化缓存命中率
- 把高频变化内容单独隔离，例如 `gitStatus`、工具列表、日期
- 把长期稳定记忆放在更容易命中缓存的位置
- 安全规则独立维护，避免和业务规则混编
- 工具指南描述“何时使用”，而不只描述“有什么能力”
- 把输出风格、效率要求当作工程约束，而不只是文案偏好

## 3. Token 管理：3 层降级压缩的完整 Pipeline

CC 的压缩设计不是“上下文超了就调一次摘要 API”，而是做成了一个带明确降级顺序的多层 Pipeline。

这部分最值得学的，不只是压缩本身，而是它把“成本”“时延”“对话连续性”这三件事一起纳入了设计。

### 3.1 为什么这样设计

长对话里的 token 管理如果处理粗糙，通常会遇到这些问题：

- 一到阈值就直接调模型做全量摘要，成本高、等待长
- 压缩发生得太晚，没有给模型留下足够生成空间
- 历史上下文被整个替换后，模型突然“失忆”
- 压缩动作阻塞用户当前交互，体验非常生硬
- 不同压缩策略混在一起，出了问题难以定位是哪一层失效

CC 的思路是先用一个明确的 buffer 提前触发压缩，再按成本从低到高逐层尝试：

- 能不调 API 就不调 API
- 能直接裁剪就不做摘要
- 能读磁盘上的预生成摘要，就不临时发起昂贵压缩
- 只有前两层都不成立时，才退回最后的全量 Legacy Compact

这套设计的目标很明确：

- 降低平均压缩成本
- 缩短用户感知延迟
- 避免突然丢失当前工作集
- 让压缩行为可预期、可解释

### 3.2 具体做法

#### 3.2.1 用固定 buffer 提前触发压缩

压缩不是等上下文窗口完全耗尽后才开始，而是提前在危险区边界触发：

```ts
// autoCompact.ts:62
const AUTOCOMPACT_BUFFER_TOKENS = 13_000

// 触发条件：当前消息占用的 token > (model_context_window - 13000)
if (currentTokens > contextWindow - AUTOCOMPACT_BUFFER_TOKENS) {
  triggerCompaction()
}
```

这个 `13,000` token 的 buffer，本质上是为模型留生成空间。也就是说，压缩系统考虑的不是“刚好还能塞下输入”，而是“压缩完之后模型还能继续稳定输出”。

#### 3.2.2 第 1 层：`MicroCompact`

第一层是每轮执行的廉价压缩策略：

- 按配置保留最近 `N` 轮消息，默认是 `10` 轮
- `N` 轮之前的 `tool result` 直接裁剪
- 最长的 message 往往正是 `tool result`，因此只裁掉老轮次就能显著控制长度
- 不调 API
- 不改变整体对话结构

实现位置是 `src/query.ts:413-426`，核心是 `createMicrocompactBoundaryMessage`。

这一层的本质不是“总结历史”，而是“先把最肥、最老、最不活跃的上下文切掉”。如果这一层就足够，系统完全不需要进入更昂贵的压缩路径。

#### 3.2.3 第 2 层：`Session Memory Compact`

如果 `MicroCompact` 之后仍然超阈值，系统进入第二层：读取预先生成好的 session 摘要，而不是临时调用模型。

这一层的逻辑是：

- 读取 `session_memory.md`
- 用 `getLastSummarizedMessageId()` 找到摘要边界
- 保留边界之后的消息
- 把摘要内容和边界后的消息重新组装成新的上下文
- 整个过程不需要再调 Claude API

实现位置是 `src/services/compact/sessionMemoryCompact.ts:514-630`。

这也是为什么有时候用户几乎感觉不到压缩发生。因为命中了已经落盘的预压缩结果，系统只是读取磁盘、重组上下文，并不是现场生成摘要。

#### 3.2.4 `session_memory.md` 不是触发时生成，而是后台异步预生成

第二层之所以快，关键在于摘要不是等触发压缩时才生成的，而是在正常对话进行中后台异步维护。

大致条件是：

- 每隔若干次工具调用触发一次检查，`toolCallsBetweenUpdates = 3`
- 并且 token 增长超过阈值，`minimumTokensBetweenUpdate = 5000`
- 这时通过 `registerPostSamplingHook` 触发
- 再由 `runForkedAgent()` 在后台提炼关键信息
- 最终写入 `session_memory.md`
- 同时记录 `lastSummarizedMessageId`

关键点在于：这里虽然也会有 API 调用，但它发生在后台对话过程中，和真正触发压缩时的用户交互解耦。

因此用户看到的体验差异往往来自这里：

- 压缩一闪而过，通常是命中了磁盘里的预压缩结果
- 压缩明显等待很久，通常是前两层都不成立，退回到最后的 `Legacy Compact`

#### 3.2.5 第 3 层：`Legacy Compact`

如果 `session_memory.md` 不存在、为空，或者第二层无法完成压缩，系统才进入第三层：

- 调 Claude API 对历史对话生成摘要
- 追加压缩边界标记 `compactBoundary`
- 未来加载时，只从边界之后恢复
- 边界之前的原始内容被丢弃

这一层是最后手段，实现位于 `src/services/compact/autoCompact.ts`。

它的成本最高，因为这里会发生一次完整模型调用，而且往往伴随较长等待。

值得注意的是，`Legacy Compact` 并不是随便生成一段自由文本摘要。CC 要求模型按结构化 JSON 输出，包括用户问题、todo 清单、澄清事项等标准字段，这使压缩结果具备：

- 结构稳定性
- 机械可处理性
- 更高的一致性

#### 3.2.6 `Preserved-tail Relinking`：压缩后仍保留当前工作集

`Legacy Compact` 里还有一个很关键的设计：压缩结果并不是只有摘要，而是：

```text
[历史摘要] + [compactBoundary 标记] + [最近 N 条活跃消息]
```

也就是除了历史摘要之外，还保留一段“最近活跃消息尾部”。

这样做的原因很实际。如果模型只看到“过去做过什么”的摘要，却看不到“刚刚正在做什么”的局部上下文，那么压缩后就很容易出现工作流断裂，表现成用户感知里的“突然忘了我在干嘛”。

`Preserved-tail Relinking` 的作用，就是在摘要之外保留当前工作集，尽量维持对话连续性。

换句话说，用户感受到的“模型忘了”，很多时候不是真的完全遗忘，而是边界截断、摘要替换和尾部保留三者之间的折中结果。问题往往出在摘要精度不足，而不是系统没有努力维持连续性。

### 3.3 可以沉淀出的最佳实践

- 压缩不要做成单一步骤，而要做成按成本降级的 Pipeline
- 触发阈值要提前留 buffer，避免压缩后没有输出空间
- 第一层优先做廉价裁剪，而不是一上来就调模型摘要
- 能用后台预生成摘要解决的问题，不要放到前台用户链路上
- 压缩结果要有明确边界标记，便于后续恢复和重建上下文
- 摘要尽量结构化输出，不要依赖自由文本
- 压缩后除了摘要，还要保留最近活跃消息，避免工作集突然断裂
- 设计 token 管理时，要同时考虑成本、时延和对话连续性，而不是只看窗口大小

## 4. 长期记忆：克制哲学与两条后台通道

很多人会觉得 CC 的长期记忆不如一些更激进的 Agent 系统，但更准确的说法不是“能力不行”，而是“默认策略更克制”。它不是尽可能多地写记忆，而是尽量避免把记忆库写乱。

这部分最值得学的，不是某个单独机制，而是它背后的记忆治理哲学：宁可少记，也不要把长期记忆做成噪声堆积场。

### 4.1 为什么这样设计

长期记忆系统如果只追求“记得更多”，通常很快会出现这些问题：

- 写入标准过宽，很多低价值内容被保存
- 记忆条目越来越多，但越来越难判断哪些仍然可信
- 冲突、重复、过时信息不断堆积
- 模型后续读取记忆时反而被噪声干扰
- 记忆系统从增强能力，变成拖慢能力

CC 的设计选择是克制。它的核心场景是代码开发，因此更看重记忆质量，而不是记忆数量。

这也是为什么很多用户会误以为它“记忆能力不强”。真正的问题不是不会记，而是默认不愿意乱记。对普通用户来说，唯一稳定可靠的触发方式通常是显式说出“记住这件事”。

这种克制不是能力缺失，而是风险控制：

- 少写错
- 少写无关信息
- 少把工作空间污染成不可维护的记忆库

相比之下，激进写入策略的优点是完整，但代价是后续一定需要更强的清理和整合机制。

### 4.2 具体做法

#### 4.2.1 默认记忆写入依赖 prompt 指令，而不是专用工具

CC 没有为普通用户暴露专门的“写记忆”工具或 Skill。它主要依赖 system prompt 里的一段长期记忆指令，大意是：

- 你拥有一个持久化的、基于文件的记忆系统
- 你应该随着时间推移持续完善这套记忆系统
- 如果用户明确要求你记住某件事，应立即保存为最合适的类型
- 如果用户要求你忘记某件事，应找到并删除相关条目

这套 prompt 实际上有两层触发逻辑：

- 模糊触发：随着时间推移主动维护记忆
- 确定触发：用户明确要求“记住某件事”

工程上真正可靠的，通常只有第二层。也就是说，如果用户不明确表达要记住，模型大概率不会主动写入。

#### 4.2.2 机制 1：`extractMemories`

`extractMemories` 是 CC 里一个被隐藏起来的主动记忆机制，需要环境变量开启，普通用户默认不可用。

它的触发时机是每轮完整 react loop 结束后的 `stop hook`。此时系统会：

- 将当前完整 `message list` fork 给一个后台 SubAgent
- 由这个 SubAgent 主动提取本轮对话里值得保留的信息
- 再写入 CC 的 memory 体系，例如 `MEMORY.md` 及相关文件

可以概括为：

```text
每轮 react 完成
  → stop hook
  → fork 完整 message list
  → 后台 SubAgent 提取长期价值信息
  → 写入记忆文件
```

这一机制的价值是把“主动识别值得记忆的内容”从主对话链路里拆出去，用户几乎无感知。

但它的代价也非常明确：一旦自动提取，系统就可能记错、记多、记下不该长期保留的内容。这正是它被设计成受控开关，而不是默认行为的原因。

#### 4.2.3 机制 2：`autoDream`

`autoDream` 是另一条后台通道，用来解决“自动写入之后如何防止记忆库劣化”的问题。它本质上是长期记忆的垃圾回收与整合机制。

它的触发特点是：

- 需要内部开关，普通用户默认不可用
- 大约每隔 `24` 小时在闲时运行
- 扫描当前工作空间中的记忆文件并做整合

整个流程分四个阶段：

```text
Phase 1: Orient
  扫描 MEMORY.md + 所有 CLAUDE.md
  识别哪些记忆需要整合

Phase 2: Gather
  收集相关上下文
  聚合碎片化信息和反复出现的主题

Phase 3: Consolidate
  合并同主题条目
  去重提炼
  覆盖写入 MEMORY.md

Phase 4: Prune
  删除过时信息、噪声、冲突条目
  保持记忆库精简干净
```

它像记忆系统的定期 GC。没有这一步，任何持续写入的长期记忆系统都很容易在时间维度上失控。

#### 4.2.4 `extractMemories` 和 `autoDream` 的分工

这两条后台通道并不是重复机制，而是分别解决不同问题：

- `extractMemories` 负责“写入”
- `autoDream` 负责“整合、去重、清理”

前者解决“哪些内容值得进入长期记忆”，后者解决“进入之后如何不变成垃圾堆”。

把两者放在一起看，CC 的长期记忆设计就很清楚了：

- 默认前台克制
- 后台可选主动提取
- 再用定期整合和清理维持长期可用性

#### 4.2.5 模型直写与后台写入有互斥保护

当用户显式要求记住某件事时，模型可以直接通过 `FileWriteTool` / `FileEditTool` 修改 `MEMORY.md`。

但模型直写和后台 `extractMemories` 不是完全并行放开的，它们之间有互斥保护，避免同时写同一文件导致竞态覆盖。

这一点很关键。因为记忆系统和普通日志不同，它是后续还要被模型反复读取和依赖的“半结构化知识库”。一旦出现并发覆盖、冲突写入，损坏成本会非常高。

### 4.3 可以沉淀出的最佳实践

- 长期记忆优先追求质量，不要先追求数量
- 默认写入策略应尽量克制，显式触发要比模糊触发更可靠
- 主对话链路和后台记忆提取链路要解耦，避免阻塞用户体验
- 自动写入记忆时，必须接受“会记错、会记多”的现实，因此要设计后续治理机制
- 长期记忆系统必须有定期整合、去重和清理流程，不能只写不管
- 把“写入”和“垃圾回收”拆成两类机制，各自独立演进
- 对记忆文件写入加互斥保护，避免竞态覆盖
- 对代码助手场景来说，混乱的记忆库往往比没有记忆更糟糕

___

## 5. 工具系统：统一接口与三层过滤

CC 的工具系统不是“模型想调什么就调什么”的自由工具箱，而是一个先统一抽象、再按上下文逐层过滤的执行体系。

这部分最值得学的有两点：

- 所有工具先收敛到统一接口，方便验证、调用和注入上下文
- 工具集不是静态常量，而是根据能力开关、运行模式和策略规则动态裁剪

### 5.1 为什么这样设计

如果工具系统没有统一抽象和过滤层，通常会很快出现这些问题：

- 每个工具的输入输出风格不同，模型调用不稳定
- 工具描述、调用方法、权限要求分散在不同实现里
- 工具和权限系统强耦合，后续难以扩展和测试
- 不同模式下哪些工具能用、不能用，边界不清晰
- 实验性工具、后台 Agent 工具、协调器工具混在一起，容易误暴露

CC 的做法是先定义统一接口，再从全量工具集出发，经过多层过滤后得到当前上下文下真正可用的工具集。

这意味着系统可以同时做到：

- 工具定义一致
- 调用协议一致
- 权限注入一致
- 模式隔离清晰
- 策略收敛到统一出口

### 5.2 具体做法

#### 5.2.1 所有工具实现统一泛型接口

CC 的所有工具都实现同一个泛型接口：

```ts
// Tool.ts — 核心接口
interface Tool<Input, Output, P extends ToolPermissionContext> {
  name: string                    // 工具名（全局唯一）
  description: string             // 模型看到的描述（影响路由决策）
  inputSchema: ZodSchema<Input>   // 输入校验（Zod）
  outputSchema: ZodSchema<Output> // 输出类型
  call(input: Input, ctx: ToolUseContext<P>): Promise<ToolResult<Output>>
  prompt: string                  // 工具使用指南（写入 System Prompt）
  permissionRule?: PermissionRule // 权限规则
}
```

这个接口设计有几个关键点：

- `name` 保证全局唯一，便于模型和系统对齐调用目标
- `description` 直接影响模型的工具路由决策
- `inputSchema` / `outputSchema` 用 Zod 统一校验和类型约束
- `call()` 接收统一的 `ToolUseContext`
- `prompt` 进入 system prompt，告诉模型“什么时候用这个工具”
- `permissionRule` 让工具可以声明自己的权限要求

#### 5.2.2 权限上下文通过依赖注入进入工具，而不是工具主动依赖权限模块

这里最值得注意的是 `ToolUseContext<P>` 里的 `P extends ToolPermissionContext`。

它意味着权限上下文是由外部注入给工具的，工具本身不直接主动引用权限模块。更具体地说：

- `query.ts` 构建权限上下文
- 通过 `context` 把权限相关信息传给工具调用
- 工具只消费上下文，不直接依赖权限系统实现

这是一种典型的依赖注入思路。它带来的好处是：

- 工具和权限系统解耦
- 工具更容易测试和复用
- 权限实现可以替换，而不影响工具接口
- query loop 可以统一控制权限边界

#### 5.2.3 从 `getAllBaseTools()` 到 `getTools()` 经过三层过滤

CC 不是直接把所有内置工具暴露给模型，而是按顺序过滤。

整体流程可以概括为：

```text
getAllBaseTools()
  → Feature Flag 过滤
  → 权限模式过滤
  → Deny 规则过滤
  → getTools()
```

其中：

- `getAllBaseTools()` 返回所有内置工具，包含实验性工具在内
- `getTools()` 返回当前上下文下真正可用的工具集

这使系统区分了“系统具备哪些工具能力”和“当前对话实际允许模型看到哪些工具”。

#### 5.2.4 第 1 层：Feature Flag 过滤

第一层过滤解决的是“这个工具在当前构建或配置里是否应被启用”。

典型场景包括：

- 某些实验性工具默认关闭
- 某些工具需要显式 feature flag 才能启用
- 协调器专用工具或 KAIROS 相关工具不应默认进入普通工具集

这一层的作用是把能力开关前置，避免尚未稳定或不适用于当前产品形态的工具被误暴露。

#### 5.2.5 第 2 层：权限模式过滤

第二层过滤解决的是“即使工具存在，也不是所有运行模式都能用”。

你给出的几类模式包括：

- `REPL_ONLY_TOOLS`
- `ALL_AGENT_DISALLOWED_TOOLS`
- `ASYNC_AGENT_ALLOWED_TOOLS`
- `COORDINATOR_MODE_ALLOWED_TOOLS`

这些规则说明工具可用性并不只取决于工具本身，还取决于当前执行主体是谁：

- 是前台 REPL
- 是普通 Agent
- 是后台异步 Agent
- 还是 Coordinator

这层过滤的核心价值，是把不同执行主体的能力边界明确分开。

#### 5.2.6 第 3 层：Deny 规则过滤

最后一层是策略规则过滤。

即使工具通过了前两层，仍然可能因为用户、项目或企业级策略的 deny 规则而被移除。也就是说，最终工具集还要再经过一层面向策略治理的收口。

这一层负责确保：

- 企业级限制可以覆盖默认能力
- 项目级规则可以限制特定工作空间行为
- 用户级配置可以进一步细化控制边界

最终得到的，才是 `getTools()` 返回给当前 query loop 的实际可用工具集。

### 5.3 可以沉淀出的最佳实践

- 先统一工具接口，再讨论工具治理
- 工具的输入、输出、描述、权限要求应统一声明
- 权限上下文优先通过依赖注入传递，而不是让工具主动依赖权限模块
- “系统拥有的工具”与“当前允许使用的工具”应明确区分
- 工具可用性至少应分为能力开关、运行模式、策略规则三层过滤
- 实验性工具和不同执行主体的专用工具要默认隔离
- 最终暴露给模型的工具集必须经过统一出口收敛，而不是在各处零散裁剪

___

## 6. 工具能力层：基础工具清单 + MCP + Skills

CC 的工具能力层不是只有一组内置工具，而是由三层能力叠加而成：

- 基础内置工具
- MCP 动态注入工具
- Skills 形式的 sub-prompt 能力扩展

这部分最值得学的，不只是“有哪些工具”，而是系统如何把“可执行工具能力”和“可复用提示能力”合并成一个完整能力集，同时保留扩展性和安全边界。

### 6.1 为什么这样设计

如果一个 Agent 系统只靠固定内置工具，通常会遇到这些限制：

- 能力边界完全由内置代码决定，扩展慢
- 第三方工具接入困难
- 某些能力其实更适合用 prompt 复用，而不是开发一个新工具
- 工具越来越多后，模型的选择难度和维护成本都会上升

CC 的做法是把能力层拆成三类：

- 对强副作用、强结构化能力，用内置工具承载
- 对外部系统接入和第三方扩展，用 MCP 承载
- 对专项方法论和可复用工作流，用 Skills 承载

这样带来的好处是：

- 基础能力稳定
- 外部扩展灵活
- prompt 级能力复用成本低
- 不同能力形态各归其位

### 6.2 具体做法

#### 6.2.1 基础内置工具按能力域分组

CC 的基础内置工具按功能可以分成几大类。

文件系统工具：

- `Read`：读取文件内容，支持分页
- `Write`：写入整个文件，新建或覆盖
- `Edit`：精确字符串替换，比 `Write` 更安全
- `Glob`：按模式匹配文件，并按修改时间排序
- `Grep`：基于 `ripgrep` 的内容搜索，支持正则
- `LS`：列出目录文件
- `NotebookRead`：读取 Jupyter Notebook，包括输出
- `NotebookEdit`：修改 Notebook Cell

Shell 执行：

- `Bash`：执行 shell 命令，支持前后台、超时和沙箱

Web 工具：

- `WebFetch`：抓取网页内容，带权限规则
- `WebSearch`：调用搜索引擎

Agent 工具：

- `Agent`：派生子 Agent 的统一入口
- `SendMessage`：向 Swarm teammate 发消息
- `TaskOutput`：读取后台 Agent 输出
- `TaskStop`：终止后台 Agent

任务管理：

- `TodoWrite`：写入或更新任务列表
- `TodoRead`：读取任务列表
- `CronCreate`：创建定时任务
- `CronDelete`：删除定时任务
- `CronList`：列出定时任务

工作模式：

- `EnterPlanMode`：进入 Plan 模式
- `ExitPlanMode`：退出 Plan 模式
- `EnterWorktree`：进入 git worktree 隔离环境
- `ExitWorktree`：退出 worktree

交互工具：

- `AskUserQuestion`：向用户提出结构化选择题
- `Skill`：调用已注册的 Skill
- `RemoteTrigger`：触发远程定时任务

这些工具构成了 CC 的基础执行能力面。

#### 6.2.2 每个工具都有独立的 `prompt.ts`，模型通过竞争式路由做选择

你给出的一个关键点是：每个工具都有独立的 `prompt.ts`。

这意味着工具系统不是简单把函数签名暴露给模型，而是给每个工具附带一段独立的使用说明。模型读完这些描述后，自主决定应该调用哪个工具。

这是一种竞争式路由设计。它的关键含义是：

- 工具描述本身就是路由策略的一部分
- 模型选错工具，不一定是推理不行，也可能是工具 prompt 写得不好
- 工具数量越多，描述质量越重要

换句话说，工具设计不仅是 API 设计，也是 prompt 设计。

#### 6.2.3 MCP 通过动态注入扩展工具能力

MCP 工具通过 `@modelcontextprotocol/sdk` 动态注入。

这意味着系统的工具能力不是编译期固定死的，而是可以在运行时接入外部能力源。MCP 在这里承担的是扩展总线的角色，让第三方能力能够进入工具层，而不必改动内置工具实现。

一个特别值得注意的点是：模板工具覆写。

如果某个 MCP 工具和内置工具同名，它可以覆盖内置工具的行为。这给第三方系统留出了非常大的扩展空间，也意味着：

- 工具能力可以被替换
- 不是所有“同名工具”都一定执行内置实现
- 运行时工具解析结果可能和基础能力层不同

#### 6.2.4 MCP 的扩展自由度必须配合文件系统安全边界

工具覆写很强大，但也带来风险。因此你给出的安全防线很关键：

- 沙箱始终阻止写入 `.claude/skills`

这项保护的重要性在于，它不只是保护一个普通目录，而是在保护一类“可影响 Agent 能力定义”的关键资源。它和保护 agents 定义文件属于同等级别的安全边界，目的就是防止恶意 MCP server 借能力注入去篡改 Skill 或 Agent 定义。

也就是说，MCP 可以扩展运行时能力，但不能随意改写系统的长期能力配置载体。

#### 6.2.5 Skills 不是工具，而是可复用的 sub-prompt 包

Skills 系统和工具系统是并列但不同形态的能力层。

Skills 的本质不是执行函数，而是“在运行时把一段结构化提示词注入当前对话”，也就是一种可复用的 sub-prompt 能力包。

你给出的优先级顺序是：

```text
bundled > plugin > user > project > flag > managed
```

运行时行为可以概括为：

```text
调用 Skill 工具
  → 展开对应 Skill
  → 作为 sub-prompt 注入当前对话
```

这说明 Skill 的价值不在于“做了一个新工具”，而在于让模型临时获得某个专项工作流、某类知识结构或某种任务套路。

#### 6.2.6 三层能力最终合并成完整能力集

从整体上看：

- 内置工具提供稳定基础能力
- MCP 提供动态扩展能力
- Skills 提供 prompt 级复用能力

三者叠加之后，才形成 AI 在当前上下文中看到的完整能力集。

这也是这张图里“金字塔”结构的意义：不是所有能力都属于同一个抽象层，而是不同层次共同构成最终可用能力面。

### 6.3 可以沉淀出的最佳实践

- 工具能力层不要只靠内置工具，应该为扩展和复用留接口
- 强副作用能力优先做成结构化工具，专项方法论优先做成 Skill
- 工具描述要当成路由策略的一部分来设计
- 第三方工具注入必须配合清晰的安全边界，尤其是对能力定义目录的保护
- 动态扩展能力和长期配置能力要严格区分，不能让运行时插件随意改写后者
- 把“执行能力”“外部扩展能力”“prompt 复用能力”拆成不同层，有助于系统长期演进

## 7. Agent 工具生态：设计范式、安全隔离与避坑指南

如果说大模型是 Agent 的大脑，那么工具就是 Agent 探索世界、改变外部状态的手和脚。真正有价值的 AI 应用，不能只停留在“会思考、会写作”，而必须能查询数据、读写文件、操作系统和调用外部服务，完成闭环任务。

但工具接入的难点，从来不只是“框架能不能调通”，而是“模型能不能看懂、用对、用稳”。这一节的重点，就是面向 Agent 的工具设计哲学。

### 7.1 为什么这样设计

很多工程团队第一次给 Agent 设计工具时，会直接把传统后端 API 的设计方式照搬过来。但传统 API 是给确定性的程序代码调用的，Agent Tool 是给基于自然语言理解的大模型调用的，两者面对的调用方完全不同。

如果没有针对 Agent 做设计调整，通常会出现这些问题：

- 工具语义过碎，模型需要多次往返拼装才能完成一个简单任务
- 参数结构过深过重，模型难以稳定生成正确输入
- 错误返回过于程序化，模型无法自我修正
- 安全上下文暴露给模型，容易被提示词注入利用
- 工具输出不受控，直接把上下文窗口打爆

因此，面向 Agent 的工具设计，不是把“已有 API 包一层壳”，而是要围绕模型的理解方式、纠错方式和上下文成本重新设计。

### 7.2 具体做法

#### 7.2.1 先区分两种主流工具调用范式

当前主流工具调用范式大致有两种：

- `Native Function Calling`
- `ReAct`

`Native Function Calling` 由模型 API 原生支持。开发者除了传入消息列表，还会传入工具列表，包括名称、描述和参数 schema。模型在理解语义后，会直接输出一个特殊结构，告诉框架要调用哪个工具以及参数是什么。

`ReAct` 则是基于：

```text
Thought -> Action -> Action Input -> Observation
```

的文本推演循环。

无论底层是原生函数调用还是 ReAct，对开发者来说，真正决定成败的不是框架接线，而是工具本身是否被设计成“模型可理解、可调用、可纠错”的形态。

#### 7.2.2 工具设计要从“传统 API 思维”切换到“面向 Agent 的思维”

这部分最核心的范式跃迁有三条。

第一，从“原子性”转向“语义完整性”。

传统 API 倾向于高内聚低耦合，把动作拆得非常原子，例如：

- `get_info_by_id(id)`
- `update_status_by_id(id, status)`

但对 Agent 来说，这种设计会让它不得不自己规划多轮组合调用，既增加错误率，也增加 token 消耗。

更合适的 Agent Tool 往往是语义完整的，例如：

- `update_user_status_by_name(name, status)`

把“查 ID -> 校验 -> 更新”的闭环收进工具内部，而不是让模型自己串三步。

第二，从“复杂强类型结构”转向“简单且可描述的结构”。

传统 API 可以接受复杂对象和深层嵌套 JSON，因为调用方是确定性的代码；但 Agent Tool 的参数越复杂，模型越容易生成错误。

因此更稳妥的做法是：

- 参数尽量扁平
- 参数类型尽量基础化
- 每个参数都要附带清晰自然语言描述

重点不是“类型够不够强”，而是“模型能不能稳定理解该填什么、不该填什么”。

第三，从“状态码报错”转向“建设性报错”。

传统 API 常见做法是返回 `error_code=1001`、`null`，或者一段底层异常堆栈。程序代码可以根据状态码走分支，但模型不行。

面向 Agent 的报错，应该直接告诉模型：

- 错在哪
- 当前输入有什么问题
- 下一步应该怎么改

例如：

```text
操作失败：时间参数格式错误，你输入了 2026/01/01，请修改为 YYYY-MM-DD 的格式后重新调用本工具。
```

这种建设性报错能直接触发模型的自我纠错，而不是把它推向幻觉或死循环。

#### 7.2.3 企业级安全调用要靠隐式上下文挂载，而不是让模型传敏感身份

在企业级环境里，工具调用涉及最敏感的问题之一，就是多租户数据隔离与身份认证。

一个典型误区是：把 `user_id` 这类敏感身份信息直接暴露成工具参数，然后期待模型在调用时老实传入当前用户的正确值。

这是不安全的，因为模型容易受到提示词注入影响，完全可能被诱导去访问其他用户的数据。

更正确的做法是：

- 敏感身份不暴露给模型
- 由系统在安全上下文里维护真实身份
- 在工具执行前通过 Hook 或拦截器注入和改写底层参数

也就是“隐式上下文挂载 + Hook 拦截”。

#### 7.2.4 用 `contextvars` 承载请求级安全上下文

你给出的示例里，`m2l8_context.py` 用 Python 的 `contextvars` 保存请求级上下文，例如：

- `user_id`
- `task_id`

这类上下文变量的关键价值是线程和协程安全。也就是说，在高并发场景下，不同用户请求的上下文不会互相串线。

这是一种很典型也很稳妥的企业级做法：把真实身份和任务上下文放进运行时安全上下文，而不是放进模型可操控的明文参数里。

#### 7.2.5 在工具调用前用 Hook 透明重写危险参数

你提供的 `m2l8_tools_call.py` 示例里，通过 `@before_tool_call` 在工具真正执行前拦截调用，并完成三件事：

- 从安全上下文里拿到真实的 `user_id`
- 构建当前用户专属的工作空间目录
- 重写工具入参中的文件路径，并做路径穿越校验

这一机制的关键价值在于：

- 模型不需要理解复杂的租户隔离逻辑
- 工具接口可以保持简单
- 底层仍然能做到严格的多租户隔离
- 即使模型尝试传入类似 `../../../etc/passwd` 的路径，也会被系统层阻断

也就是说，安全控制应该发生在工具执行边界，而不是寄希望于模型“自觉守规矩”。

#### 7.2.6 典型反模式：全能工具、深嵌套参数、沉默失败和上下文炸弹

你给出的几类反模式都很典型。

第一，全能工具（God Tool）。

例如写一个 `DatabaseManager`，再让模型通过 `action_type` 去决定是 `insert`、`delete`、`query` 还是 `update`。这种设计会严重破坏单一职责，也会让模型很容易搞混不同动作的参数约束。

第二，多层转义参数和深嵌套入参。

例如某个参数要求传 JSON 字符串，而这个 JSON 字符串内部某个字段又是另一段 JSON 字符串。或者结构是 `Map -> List -> Map` 这种深层嵌套。对模型来说，这类参数的错误率会迅速上升。

第三，沉默失败（Silent Failure）。

如果工具失败时只返回空字符串、`null` 或一段原始异常堆栈，模型并不知道哪里错了，也不知道该怎么修复，最终往往会反复重试同样错误。

第四，上下文炸弹（Context Bomb）。

如果一个工具一次性返回几万行数据、几十兆文件内容，虽然“信息很全”，但会直接撑爆上下文窗口，或者让模型被海量无关细节淹没，忘掉当前任务目标。

#### 7.2.7 面向 Agent 的工具最佳实践

对应前面的反模式，更稳妥的做法是：

- 用建设性报错替代状态码和空返回
- 给大数据量工具设计分页或摘要机制
- 通过分页参数引导模型逐页获取内容
- 必要时先在工具内部做摘要压缩，再返回主 Agent

这一点很重要：工具不只是“把底层系统原样暴露给模型”，而是要主动替模型做信息整形、风险隔离和错误解释。

### 7.3 典型反模式

- 把一个工具设计成“全能入口”，再通过 `action_type` 让模型自己决定分支，会显著增加参数混淆和调用错误
- 让模型生成多层 JSON 字符串、深嵌套对象或复杂转义结构，会迅速抬高工具调用失败率
- 工具失败时返回空字符串、`null` 或原始异常堆栈，会让模型无法自我纠错，只会盲目重试
- 一次性把海量原始结果直接返回给模型，会制造上下文炸弹，既浪费 token 又干扰推理
- 把 `user_id`、租户标识或真实文件路径直接交给模型生成，是企业级工具调用里最危险的越权入口

### 7.4 可以沉淀出的最佳实践

- 工具设计应面向模型理解方式，而不是照搬传统后端 API
- 优先提供语义完整的工具，而不是让模型拼装多个原子调用
- 参数结构尽量扁平，参数描述尽量具体
- 错误返回必须是可操作的自然语言指导，而不是状态码
- 敏感身份和租户信息不要暴露给模型作为显式参数
- 在工具执行边界注入安全上下文，比在 prompt 里强调规则更可靠
- 对文件、数据库、外部系统访问都应默认做路径校验、租户隔离和越权拦截
- 大结果集工具必须设计分页、摘要或裁剪机制，避免上下文炸弹
- 反模式识别也应纳入工具评审标准，例如 God Tool、深嵌套参数和沉默失败

___

## 8. 从传统 API 到 Agent Tool：五步标准 SOP

企业里真正常见的情况不是“从零设计一套完美的 Agent 工具”，而是面对已经存在的大量传统 API，思考如何把它们平滑改造成大模型能稳定使用的工具层。

这部分最有价值的地方，不是讲一个抽象原则，而是给出一套可复用的封装 SOP。只要底层系统接口还在，你就可以按这套方法把它们逐步改造成面向 Agent 的能力。

### 8.1 为什么这样设计

传统业务 API 大多是面向前端页面、后端服务或者内部系统集成设计的，因此天然会带有这些特点：

- 接口语义高度原子化
- 输入输出结构复杂，系统字段多
- 错误返回偏程序化
- 参数名和字段含义依赖业务背景知识
- 返回体包含大量对模型无意义的元数据

如果把这些 API 原样暴露给 Agent，通常会出现这些问题：

- 模型需要自己串联多步调用才能完成一个业务动作
- 输入参数容易填错，尤其是复杂嵌套结构
- 错误发生后模型无法理解如何修复
- 输出冗余太多，干扰推理甚至打爆上下文

所以，传统 API 到 Agent Tool 的改造，本质上不是“换个 SDK”，而是一次面向模型使用方式的接口重构。

### 8.2 具体做法

#### 8.2.1 Step 1：语义完整性重构

第一步不是写代码，而是重做接口抽象。

传统后端接口往往是原子化的，例如：

- `get_user_by_name`
- `check_permission`
- `update_user_info`

从程序设计看，这种拆分没问题；但从 Agent 使用看，这意味着模型必须自己规划并执行完整调用链。

更适合 Agent 的做法，是在工具层进行聚合，提供一个具备完整业务语义的接口，例如：

- `update_user_info_by_name`

然后在工具内部用确定性代码去完成：

- 查用户
- 校验权限
- 执行更新

也就是说，让模型负责“理解意图”，让代码负责“稳定流转”。

#### 8.2.2 Step 2：I/O 瘦身

第二步是把传统 API 的臃肿输入输出压缩成 Agent 真正需要理解的形态。

输入侧的问题通常是：

- 嵌套层级深
- 系统级字段多
- 鉴权、路由、资源类型等细节暴露给模型

例如一个传统搜索 API 可能会要求模型理解：

- `messages`
- `search_source`
- `resource_type_filter`
- `search_filter`
- 深层的 `match` 和 `query.filter.range`

但对 Agent 来说，真正核心的业务输入通常只有：

- `query`
- `top_k`
- `site`
- `recency_filter`

因此输入瘦身的目标是：

- 去嵌套
- 去冗余
- 去系统内部字段
- 保留业务语义最强的核心参数

输出侧同样需要瘦身。传统 API 返回体里经常包含很多无关字段，例如：

- 内部 `request_id`
- `icon: null`
- `image: null`
- 对模型推理无帮助的内部标识字段

更适合 Agent 的输出，应该只保留：

- 结果标题
- 链接
- 摘要
- 条目数量

并尽量使用带自然语言解释的格式，让模型知道每个字段的含义，而不是只给裸 JSON。

#### 8.2.3 Step 3：错误提示要变成建设性错误

第三步是错误处理改造。

对 Agent 来说，工具错误不是“出异常就算了”，而是一次自我修正机会。工具层应该把底层异常转成面向模型的自然语言反馈，至少回答三件事：

- 错误是什么
- 原因是什么
- 下一步应该怎么改

你给出的百度搜索案例里，错误处理就是按这个思路写的。对于不同错误码，会映射成更友好的错误描述，例如：

- 参数错误
- 服务超时
- API Key 认证失败

然后再组合成结构化自然语言输出：

```text
错误：API 返回错误。
原因：……
解决提示：……
```

这种错误信息能直接激活模型的自我纠错，而不是让它在不理解错误的情况下反复重试。

#### 8.2.4 Step 4：工具描述要承担触发路由职责

第四步是工具描述设计。

工具描述不只是“这是什么工具”，更重要的是告诉模型：

- 什么时候该用它
- 调用后会得到什么
- 什么场景不该用它

这里有两个关键点。

第一，工具名要规范，最好使用“动词 + 名词”。

例如：

- 不推荐 `Baidu_Search`
- 更推荐 `web_search`

因为在很多框架里，工具名本身就是最直接的动作提示。

第二，工具描述最好至少包含四类信息：

- 功能描述
- 产出物说明
- 触发时机
- 适用边界

你给出的百度搜索案例里，`search_web` 的 `description` 就明确写了：

- 能做网络搜索
- 可以按时间范围和站点筛选
- 会返回标题、链接和摘要
- 适用于查找公开信息
- 当存在更专业工具时，不优先使用它

这类描述的价值，是减少模型在“该不该调用这个工具”上的逻辑负担。

#### 8.2.5 Step 5：参数描述也要 Prompt 化

第五步是参数描述模板化。

不要只给模型一个参数名，比如 `top_k`，让它自己猜。参数描述应该直接承担“使用说明书”的角色。

一个高质量参数描述，至少应包含：

- 参数含义
- 取值边界
- 默认值
- 典型业务场景下的赋值建议

你给出的百度搜索案例就做得比较完整。

例如 `top_k`：

- 默认 `20`
- 精确信息搜索时推荐 `5` 以下
- 广泛调研时建议 `10` 以上

例如 `recency_filter`：

- 可选值有哪些
- 各自对应的时间范围
- 什么场景适合加时间过滤
- 什么场景不适合加时间过滤

例如 `sites`：

- 最多支持多少个站点
- 示例格式是什么
- 不同任务通常该选哪些权威站点

这些描述的本质，不是在“补文档”，而是在直接教模型如何更稳定地赋值。

#### 8.2.6 百度搜索案例展示了完整 SOP 是如何落地的

你提供的 `BaiduSearchTool` 示例，实际上正好对应这套五步 SOP：

- 用 `BaiduSearchInput` 定义输入 schema
- 用 `BaseTool` 封装统一工具接口
- 在 `_run()` 中完成底层参数映射
- 把传统 API 错误转换成建设性错误
- 把原始搜索结果格式化成便于 Agent 理解的文本结果

其中几个特别值得注意的细节是：

- `query`、`top_k`、`recency_filter`、`sites` 都有较强的自然语言描述
- 工具名用了 `search_web` 这种更贴近触发语义的形式
- `description` 写明了触发时机和适用边界
- 输出不是原始 JSON，而是带标题、链接、摘要的结果列表

这说明真正优秀的 Agent Tool 封装，不只是“能调用 API”，而是要在工具层把输入理解、底层映射、错误修复和输出解释一起做好。

### 8.3 可以沉淀出的最佳实践

- 不要把传统 API 原样暴露给 Agent，要先做语义重构
- 工具应该尽量提供目标导向的闭环能力，而不是只暴露原子动作
- 输入参数尽量拍平，输出结果尽量去噪
- 工具层负责把复杂底层协议映射成模型易懂的简单接口
- 错误返回要可复盘、可修复，而不是只给状态码
- 工具名、描述、参数说明都应视为 prompt 设计的一部分
- 参数说明尽量写出默认值、边界和业务场景，而不是只写字面含义
- 结果输出优先做成对模型友好的摘要化文本，而不是直接返回臃肿结构体

## 9. MCP：协议抽象、生态复用与企业级接入

MCP（Model Context Protocol）是当前 Agent 工具生态里最关键的标准化协议之一。它真正改变的，不只是“怎么接一个工具”，而是把 AI 应用和外部系统之间的关系，从手工硬编码适配，升级成了协议驱动的能力发现与调用。

这一节最值得学的，不是某个框架怎么写配置，而是 MCP 为什么会成为企业级 Agent 工具接入的基础设施。

### 9.1 为什么这样设计

如果没有 MCP 这类协议层，AI 应用接入外部系统通常会遇到这些问题：

- 每接一个新系统都要写一套专用适配代码
- 同一个工具能力无法在不同 Agent 框架和客户端之间复用
- AI 团队和后端团队之间耦合过深，协作成本高
- 工具发现、schema 描述和调用协议都不统一
- 不同系统的安全、监控和鉴权方案难以收敛

MCP 的核心思想是：它首先是一个协议，而不是一个具体的软件产品。

就像 HTTP 让浏览器和服务器能够按标准通信一样，MCP 让：

- MCP Client
- MCP Server

之间也能按标准交换工具能力描述、参数 schema 和调用结果。

这意味着只要某个外部系统把自己的能力按 MCP 协议暴露出来，AI Client 就可以：

- 自动发现工具
- 自动理解工具 schema
- 动态调用工具

而不必在 Agent 代码里硬编码具体业务逻辑。

### 9.2 具体做法

#### 9.2.1 先明确 MCP 的 Client / Server 分工

MCP 把工具调用链路拆成了两端：

- `MCP Client`
- `MCP Server`

其中：

- `MCP Client` 是 Agent 或 AI 应用，负责自然语言理解、推理和决策
- `MCP Server` 是底层工具和数据源的暴露端，例如邮箱系统、数据库、OA 系统等

这层分工的意义在于，Agent 不再需要直接理解每个业务系统的细节，而是统一通过 MCP 这套标准协议消费能力。

#### 9.2.2 MCP 的核心价值是生态复用与架构解耦

MCP 在企业落地里的两个核心价值非常明确。

第一，生态复用。

只要一个系统被封装成标准 MCP Server，那么：

- 今天可以被 CrewAI 调用
- 明天也可以被 LangChain 应用调用
- 未来也可以被 Claude 桌面端、Cursor 或其他支持 MCP 的客户端直接接入

这本质上就是“Write Once, Use Anywhere”。

第二，架构解耦。

在大型企业里：

- AI 团队负责大脑和应用编排
- 后端团队负责业务系统和工具服务

有了 MCP 后，后端团队只需要按协议暴露能力，AI 团队就能直接消费，不需要两边长期绑定在某个框架实现细节上。

#### 9.2.3 当前 MCP 生态的资源入口和分布特点

你给出的生态现状可以概括成两部分。

先看资源入口，主要有四类：

- 官方和第三方资源站，例如官方 registry、Smithery.ai、MCP.so
- AI 客户端内置商店，例如 Cursor、Cline、Glama、Windsurf
- 托管基站，例如 Composio、Val Town
- 开源社区资源列表，例如 `awesome-mcp-servers`

再看能力分布，当前生态明显更偏开发者导向：

- 开发者工具占大多数，例如 git、数据库、流水线、排障
- 通用 AI 工具也在快速增长，例如浏览器、搜索、文档处理
- 大厂 API 正在逐步 MCP 化，例如地图、即时通讯、文档平台
- 专业垂类领域还在起步阶段

这意味着：MCP 已经进入可用阶段，但生态成熟度在不同领域差异仍然很大。

#### 9.2.4 自研 MCP Server 应作为企业对外暴露工具能力的标准载体

如果企业想把内部业务系统开放给 AI 调用，最稳妥的方式不是让 Agent 直接打散乱的内部 API，而是开发自己的 MCP Server。

你给出的 FastAPI 模板框架体现了几个企业级特征：

- 支持新的 Streamable HTTP 传输协议
- 基于 ASGI 异步架构
- 内置结构化 JSON 日志
- 内置 Prometheus 监控指标
- 内置 API Key 鉴权
- 通过装饰器模式完成工具注册

这说明一个好的 MCP Server，不只是“把函数暴露出去”，还应该同时具备：

- 可观测性
- 鉴权能力
- 并发能力
- 工具注册和维护的工程化效率

#### 9.2.5 自研 MCP Server 的开发重点仍然是工具描述与参数说明

即使工具通过 MCP 对外暴露，真正决定 Agent 调用稳定性的，仍然是工具本身的说明质量。

你给出的开发重点包括：

- 规范工具名，使用“动词 + 名词”
- 写清触发时机
- 写清适用边界
- 参数描述写明边界、默认值和示例

这说明 MCP 并没有替代前面几节课讲过的工具设计原则，它只是把这些原则放进了一个标准化协议壳里。

也就是说：

- MCP 解决的是“怎么标准化暴露工具”
- 工具描述质量解决的是“模型能不能稳定调用”

两者不是替代关系，而是叠加关系。

#### 9.2.6 密钥管理和身份映射必须留在 Server 端

你给出的邮件 MCP Server 案例里，有一个非常重要的企业级安全原则：

- 绝不能让 LLM 直接管理密钥

正确做法是：

- 用户密钥在 Server 端加密存储，例如用 Fernet
- Agent 调用时通过 HTTP Header 传递 `X-User-Id`
- Server 根据用户 ID 去换取真实密钥并执行业务逻辑

这样模型看到的只是：

- 一个 URL
- 一组安全信道里的 header

而不会接触到底层真正敏感的授权码、私钥或令牌。

#### 9.2.7 在 Client 侧接入 MCP 的关键不是“连上”，而是“过滤”

在 CrewAI 这类框架里，MCP Client 接入很简单，只需配置 `MCPServerHTTP`，提供：

- `url`
- `Authorization`
- `X-User-Id`
- `tool_filter`

这里最关键的安全机制不是连接本身，而是 `tool_filter`。

你给出的 `create_static_tool_filter` 思路很重要：

- 即使远端 MCP Server 后续新增了一百个工具
- Agent 也只能看到白名单里的那几个

这避免了远端服务变更、恶意工具注入或工具面过大带来的不可控风险。

#### 9.2.8 MCP 工具注入本质上是一次动态发现、过滤和 schema 拼装

当 Agent 运行时，底层实际上会经历这样一个流程：

```text
1. 连接 MCP Server
2. 调用 tools/list 发现工具
3. 经过本地 filter 做安全过滤
4. 将保留下来的工具 schema 组装进模型上下文
5. 让模型像使用本地工具一样调用远程能力
```

有些框架甚至会给远程工具生成类似：

- `http_localhost_8005_mcp_send_email`

这样的内部名字，然后注入到 system prompt 中。

从模型视角看，它几乎感觉不到“这是远程服务”，因为协议层已经把远程能力伪装成了一个标准工具。

#### 9.2.9 MCP 的典型反模式

你给出的几个反模式都非常关键。

第一，巨型 MCP。

把过多复杂工具塞进一个 MCP Server，一次暴露二十多个工具，直接让工具 schema 吃掉大量 token。这不仅浪费成本，还会分散模型注意力。

第二，过度原子化。

如果一个任务需要模型通过 MCP 连续做五六次细碎调用，失败概率和成本都会显著上升。MCP 服务提供的工具，仍然应该遵守语义完整性原则。

第三，同步阻塞时间过长。

在 Stream / SSE 长连接模式下，如果后端工具执行长时间重 I/O 操作却没有心跳，Client 可能直接超时断开。

第四，把系统安全参数交给模型。

例如让模型自己传 `user_id`。这会直接引入严重越权风险。

#### 9.2.10 MCP 服务要按无状态和幂等原则设计

在最佳实践里，你给出的两条非常重要。

第一，尽量无状态。

MCP Server 应尽量只负责能力执行，而把业务上下文和任务进度留给 Client 管理。这有助于：

- 扩展部署
- 减少服务端复杂度
- 让状态边界更清晰

第二，必须考虑幂等。

由于网络抖动、SSE 不稳定或 Client 自动重试，某个请求可能被重复发送。如果工具本身不是幂等的，那么：

- 可能重复发送邮件
- 可能重复扣款
- 可能重复触发关键操作

因此，核心操作型工具应支持类似 `request_id` 的幂等键。

### 9.3 典型反模式

- 把大量复杂工具无节制塞进同一个 MCP Server，会形成“巨型 MCP”，占满 schema token 并干扰模型路由
- 把工具拆得过细，导致模型需要多次细碎往返调用，既增加成本也放大失败概率
- 长耗时 MCP 工具没有心跳、流式反馈或超时治理，容易在 Stream/SSE 模式下被 Client 误判断开
- 把 `user_id`、权限范围、租户信息等安全参数交给模型生成，会直接破坏权限边界
- 对关键副作用操作不做幂等设计，一旦网络重试就可能重复发邮件、重复扣款或重复执行危险动作

### 9.4 可以沉淀出的最佳实践

- 把 MCP 当成协议层，而不是某个框架私有能力
- 优先用 MCP 解耦 AI 应用与底层业务系统
- 企业内部工具接入应优先建设自研 MCP Server，而不是直接暴露内部 API
- MCP Server 除了工具逻辑，还应具备日志、监控、鉴权等基础设施
- 工具 schema 的描述质量依然决定模型调用成功率
- 密钥和敏感身份信息必须留在 Server 端，通过安全信道传递上下文
- Client 侧必须加白名单过滤，不能盲目信任远端工具集
- 避免巨型 MCP 和过度原子化，保持工具集克制且语义完整
- 对长耗时工具考虑流式心跳、超时和断连问题
- 对关键副作用操作必须做幂等设计

___

## 10. 执行能力层：文件操作、代码解释器与无头浏览器

如果说 MCP 解决的是“怎么接入外部能力”，那么这一节解决的是“Agent 拿到能力之后，如何真正完成复杂闭环任务”。从工程视角看，文件操作、代码解释器和无头浏览器，是把 Agent 从会聊天、会规划，推进到能真正执行任务的三大核心能力。

但这三类能力一旦开放，也意味着系统开始进入高风险区。能力越强，越需要沙箱、隔离和执行优先级设计。

### 10.1 为什么这样设计

一个只能调用简单工具、不能处理文件、不能运行代码、不能理解网页交互的 Agent，本质上仍然像一个高级问答系统，而不是真正的“数字白领”。

要完成复杂现实任务，Agent 至少需要三类递进能力：

- 文件操作能力
- 代码解释器能力
- 无头浏览器能力

这三类能力分别解决不同问题：

- 文件操作负责对本地工件进行持久化和精细修改
- 代码解释器负责把复杂计算、数据处理和程序化抓取交给代码完成
- 无头浏览器负责在没有标准 API 时，像人一样操作网页

但与此同时，如果这些能力直接在宿主机上裸奔运行，风险也会急剧放大：

- 模型幻觉可能生成毁灭性命令
- Prompt Injection 可能诱导模型下载恶意脚本
- 高权限执行可能让宿主机直接沦陷

因此，这一节的核心不是“给 Agent 更多权力”，而是“在受控沙箱内给 Agent 正确的执行权”。

### 10.2 具体做法

#### 10.2.1 三大王牌超能力各自解决什么问题

第一类是文件操作能力。

Agent 不只是需要：

- `Read`
- `Write`

更关键的是需要：

- `Edit`
- `Diff`

也就是局部编辑能力。因为面对大型代码文件、长报告或复杂配置时，Agent 更常见的需求不是全量重写，而是做精确局部修改。这个能力直接决定了它在代码重构、文档修订和复杂编辑任务中的稳定性。

第二类是代码解释器。

它让 Agent 在遇到复杂问题时，不再依赖纯“脑补”，而是像程序员一样：

- 写代码
- 运行代码
- 读取结果
- 再继续推理

这使它能处理：

- 数学计算
- 表格清洗
- 数据统计
- 图表生成
- 程序化抓取

第三类是无头浏览器。

当目标系统没有现成 API，或者信息只存在于动态网页里时，无头浏览器就是终极执行手段。它赋予 Agent：

- 打开网页
- 滚动页面
- 点击元素
- 获取截图
- 结合视觉能力理解页面状态

#### 10.2.2 绝不能让这些能力在宿主机上裸奔

你给出的安全警告是这一节最重要的前提。

如果代码解释器或浏览器执行直接运行在宿主机上，风险非常具体：

- 幻觉命令可能误删关键文件
- 恶意 prompt 可能诱导模型执行挖矿或木马脚本
- 一旦对外暴露，主机可能被迅速利用成攻击跳板

因此，执行环境必须和宿主机隔离。这里不是“最好隔离”，而是“必须隔离”。

#### 10.2.3 用沙箱把高风险执行能力关进笼子

工业界常见做法是使用 Sandbox 隔离技术，把 Agent 的高风险执行能力放进一个受控容器里。

你给出的推荐方案是 AIO-Sandbox。它基于 Docker 容器，把：

- 写代码
- 运行依赖
- 浏览器操作
- 文件处理

这些能力都放进一个独立环境中执行。

这样即使 Agent：

- 把系统搞崩
- 装错依赖
- 执行危险脚本

影响也被限制在沙箱内部，只需重启容器即可恢复。

#### 10.2.4 通过 MCP 把沙箱能力暴露给外层 Agent

这套架构的关键点在于，沙箱不是直接被 Agent 以本地函数形式调用，而是通过 MCP 对外暴露能力。

这意味着：

- 沙箱本身是能力提供方
- Agent 通过 MCP 获取这些能力
- 外层编排逻辑和底层执行环境解耦

你给出的例子里，沙箱可以暴露出：

- `sandbox_browser_screenshot`
- `browser_click`
- `browser_scroll`
- `browser_get_clickable_elements`

等大量浏览器原语，以及运行代码的能力。

这实际上就是把“全能数字实验室”包装成一个标准化的远程工具服务。

#### 10.2.5 复杂任务中要让 Agent 交替使用代码与浏览器能力

阿里股票早报的案例，体现了这三类能力如何在一个真实任务里组合使用。

在这个任务中，Agent 的目标不是单纯“搜点信息”，而是完成一个结构化输出的复合任务：

- 获取最新行情和 30 日 K 线数据
- 用 Python 做量化分析
- 用浏览器搜索舆情
- 最终输出结构化报告

这个过程中，Agent 会在沙箱里：

- 写 Python 代码获取和分析金融数据
- 调用浏览器去搜索和收集新闻
- 最终整合成报告

这说明一旦执行环境搭好，Agent 的能力边界就不再只取决于模型本身，而取决于它能调度哪些可靠执行能力。

#### 10.2.6 这类能力最常见的反模式

你给出的几个反模式都非常典型。

第一，主机裸奔。

这是最致命的问题，高风险执行能力绝不能直接运行在宿主机上。

第二，被污染的 `stdout`。

代码解释器的输出通常会回流给模型。如果执行结果里混进大量日志、警告、框架噪声，或者代码根本没有把关键结果 `print()` 出来，模型就会拿到一堆无效信息，最终反复重试。

第三，生吞 HTML。

如果浏览器工具把网页原始 HTML 整块返回给模型，里面会有大量：

- `<div>`
- `<script>`
- CSS
- 框架残留

这会迅速吃光 token，还会淹没模型真正该关注的内容。

第四，忽视 SPA 异步渲染。

现代网页大量采用 Vue、React 等 SPA 架构。如果页面刚打开就立刻抓取，Agent 很可能只能看到 loading 状态或尚未加载完成的空壳。

#### 10.2.7 稳定落地时要定义执行优先级和环境约束

这部分的最佳实践里，有几个特别关键。

第一，明确工具优先级。

应该在 prompt 或 backstory 里明确告诉 Agent：

- 代码调用 API / 第三方包
- 优先于专业搜索工具
- 再优先于浏览器硬爬

原因很简单：程序化获取结构化数据，通常比浏览器手动扒页面更稳定、更快、更便宜。

第二，预置依赖。

在 Docker 沙箱镜像里预装常见 Python 库，例如：

- `pandas`
- `requests`
- `beautifulsoup4`

并在提示词里明确告知模型这些库已经存在，避免它反复尝试 `pip install`。

第三，浏览器潜行伪装。

当不得不抓网页时，要尽量降低被反爬机制拦截的概率，包括：

- 伪装 User-Agent
- 使用合理屏幕分辨率
- 尽量接近真实用户行为

第四，必须使用足够强的模型。

代码解释器和浏览器联动任务，对规划能力要求很高。如果模型本身太弱，它只会在沙箱里持续撞墙，把错误和重试成本全部放大。

### 10.3 典型反模式

- 让代码解释器或浏览器执行直接跑在宿主机上，是最危险的执行架构错误
- 代码执行结果的 `stdout` 混入大量日志、警告或无关噪声，会把模型带进无效重试循环
- 直接把原始 HTML 整块塞回模型，会迅速抽干上下文预算并淹没真正有价值的信息
- 抓取现代 SPA 页面时忽略异步渲染和等待机制，通常只能得到 loading 空壳
- 在本可通过 API 或程序化方式获取数据时，仍让 Agent 用浏览器硬爬，会显著降低稳定性和效率

### 10.4 可以沉淀出的最佳实践

- 文件操作、代码解释器、无头浏览器是 Agent 进入复杂执行场景的三大核心能力
- 高风险执行能力必须运行在受控沙箱中，不能直接运行在宿主机上
- 沙箱能力更适合通过 MCP 或等价协议对外暴露，保持执行层与编排层解耦
- 对复杂任务要明确工具优先级，优先程序化获取数据，再考虑浏览器硬爬
- 代码解释器的输出必须可读、可控，避免噪声污染模型上下文
- 浏览器抓取不要直接生吞原始 HTML，应优先抽取结构化信息或页面快照
- 处理现代网页时必须考虑异步渲染和等待机制
- 预置常用依赖能显著提升 Agent 执行稳定性和效率
- 强执行能力必须配合足够强的模型，否则只会放大失败成本

___

## 11. Skills：面向 Agent 的知识封装与执行引擎

当 Agent 已经拥有了工具、MCP 和沙箱能力之后，新的问题就变成了：它知道“怎么正确地用这些能力”吗？

很多最佳实践不是工具 schema 能表达的。例如处理 PDF 时，先检测文字层，再决定是直接提取还是走 OCR；如果有密码保护要先停下来询问；有些中文 PDF 需要特殊处理。模型并不会天然知道这些经验。

Skills 的意义，就是把这些经验变成 Agent 可以读取、理解、执行和复用的结构化操作手册。

### 11.1 为什么这样设计

如果没有 Skills，企业在把专家经验交给 Agent 时通常会遇到三类问题。

第一，上下文爆炸。

如果你想让 Agent 具备二十种文档处理能力，就需要把二十份操作说明全塞进 system prompt。每份说明都很长，总量很快会挤爆上下文窗口，即使勉强塞进去，也会因为相互干扰导致模型注意力分散。

第二，经验难以沉淀和复用。

某个团队在 prompt 里写了一堆最佳实践，换一个 Agent、换一个项目、换一个团队，这些知识又得重新整理。知识只能活在 prompt 字符串里，无法像代码一样治理、复用和版本化。

第三，工具有了，但不会正确使用。

MCP 和工具 schema 解决的是“Agent 能做什么”，却没有解决“Agent 在什么场景下按什么步骤做才对”。JSON Schema 可以描述参数，却无法完整承载人类专家积累的操作经验。

因此，Skills 的核心价值不是增加一种新工具形态，而是给 Agent 增加一层“知识执行层”。

### 11.2 具体做法

#### 11.2.1 Skills 的本质是结构化操作手册，而不是 API

首先要明确，Skills 不是：

- API
- MCP Server
- 插件格式

Skills 的本质，是“给 LLM 读的结构化操作手册”。

一个完整的 Skill 通常由这些部分组成：

```text
skill-name/
├── SKILL.md
├── scripts/
├── references/
└── assets/
```

其中：

- `SKILL.md` 是核心说明书
- `scripts/` 放可执行脚本
- `references/` 放参考文档
- `assets/` 放静态资源

`SKILL.md` 本身又分两层：

- YAML frontmatter：给主 Agent 看，用于做路由决策
- Markdown 正文：给执行层看，用于说明具体步骤和硬性规则

例如 `CRITICAL`、`NEVER`、`ALWAYS` 这类规则块，本质上是在把专家经验显式编码给模型。

#### 11.2.2 Skills 背后的哲学是 “Bash is All You Need”

Skills 和传统 MCP 工具的最大区别在于：

- MCP 更倾向于预先把能力封装成结构化 API
- Skills 更倾向于把“怎么做”写成操作说明，再让 Agent 现场写代码执行

这背后对应的是一种很强的工程哲学：

- 只要给 Agent 基础执行能力，例如 `read`、`write`、`edit`、`bash`
- 再给它正确的操作说明
- 它就可以在长尾场景里自己生成代码、自己调试、自己完成任务

这也是为什么 Skills 特别适合承载那些很难预先穷举为 JSON Schema 的经验性流程。

#### 11.2.3 Skills 分为参考型和任务型

你给出的执行架构里，Skills 至少可以分为两类：

- 参考型 Skill
- 任务型 Skill

参考型 Skill 比较轻量，主要作用是把一套规范或知识注入主 Agent，让它自己按规则处理问题。

任务型 Skill 更重，会真正调起沙箱和执行能力，在独立上下文里完成一个子任务。

这一区分非常重要，因为它决定了 Skill 是：

- 只提供知识约束
- 还是要启动一个带工具和沙箱的子执行引擎

#### 11.2.4 渐进式披露是 Skills 可扩展的核心机制

Skills 能大规模扩展而不把主 Agent 上下文撑爆，关键在于渐进式披露（Progressive Disclosure）。

它的核心思路是分两阶段加载。

第一阶段，系统启动时轻量加载。

这里只读取每个 `SKILL.md` 的 frontmatter，提取：

- `name`
- `description`

然后把它们组装成轻量描述，例如 XML，注入到 `SkillLoaderTool` 的工具描述里。此时主 Agent 只知道：

- 有哪些 Skill
- 各自适合什么场景

而不会读完整操作手册。

第二阶段，真正调用某个 Skill 时按需加载。

只有当主 Agent 决定使用某个 Skill 时，系统才会：

- 读取完整 `SKILL.md`
- 剥离 frontmatter
- 拼接沙箱约束
- 注入 Skill Agent 的 backstory
- 启动独立 Sub-Crew 执行

执行结束后，Sub-Crew 的上下文销毁，主 Agent 只收到结构化结果。

这使系统满足两个关键目标：

- Skills 库可以无限增长
- 主 Agent 上下文始终保持干净

#### 11.2.5 `SkillLoaderTool` 是渐进式披露的核心实现

你给出的 `SkillLoaderTool` 实现里，有几个设计值得重点记录。

第一，初始化时构建轻量 description。

`__init__()` 会读取 `load_skills.yaml`，遍历启用的 Skill，解析 `SKILL.md` frontmatter，然后把名称、类型、描述拼成 XML 注入工具 description。这就是渐进式披露的第一阶段。

第二，完整指令按需缓存。

`_get_skill_instructions()` 在第一次调用某个 Skill 时，读取完整 `SKILL.md` 正文，去掉 frontmatter，再拼接沙箱执行约束，并缓存结果。后续再调用同一 Skill 时，可以直接命中缓存。

第三，路径约束被硬编码进 sandbox directive。

这点很关键，因为它直接减少了模型路径幻觉的空间。不是让模型自由推测脚本在哪，而是明确告诉它：

- Skill 资源挂载在什么位置
- 应该用什么命令执行
- 缺依赖时如何处理

#### 11.2.6 这里有几个很关键的工程细节

你给出的代码里，有几个细节非常典型。

第一，`PrivateAttr`。

像 `_skill_registry`、`_instruction_cache` 这类字段，如果不通过 `PrivateAttr` 声明，而是直接写成普通类属性，在 Pydantic V2 里很容易被当成类级共享字段，或者直接引发校验问题。

第二，`field_validator(mode="before")`。

`task_context` 在实际 LLM function call 里，有时会传来 `dict` 或 `list`，而不是字符串。通过前置 validator 统一转字符串，可以显著减少工具调用时的 schema 校验失败。

第三，把约束写进 `Field.description`，而不是只写进 Agent backstory。

这样这些约束会进入工具 schema，模型在构造 function call 时能更精准地看到要求，而不是在长 prompt 里被稀释。

#### 11.2.7 任务型 Skill 依赖独立的 Sub-Crew 工厂

任务型 Skill 的执行不是主 Agent 直接完成，而是由 `build_skill_crew()` 这样的工厂函数，每次构造一个全新的 Skill Sub-Crew。

这里的关键设计有两个。

第一，必须每次构造新 Crew。

不能复用旧实例，因为上一次执行残留的 Task 状态会污染下一次。

第二，要对白名单工具做最小权限过滤。

即使沙箱暴露了几十个工具，Skill Agent 真正可见的也应该只是少数几个必需工具，例如：

- 执行 bash
- 执行代码
- 文件读写
- 字符串替换编辑

这既是安全策略，也是性能优化。因为工具越多，schema 注入越重，模型的选择负担越高。

#### 11.2.8 主 Agent 负责编排，Skill Agent 负责专门执行

这套架构里，主 Agent 不是去亲自执行 PDF、DOCX 等复杂流程，而是扮演协调者。

它的职责包括：

- 理解用户需求
- 规划子任务
- 调用 `skill_loader`
- 检查子任务返回的 `errcode` / `errmsg`
- 汇总最终结果

而真正执行某个专业流程的，是 Skill Sub-Crew。

这种设计的价值在于：

- 主 Agent 保持轻量编排角色
- 专业操作逻辑被隔离到专门 Skill
- 执行上下文和主上下文天然分离

#### 11.2.9 Skills 和 MCP 的关系是“知识层 + 能力层”

Skills 和 MCP 不是简单替代关系，而是更复杂的合作与竞争关系。

它们的天然分工是：

- MCP 解决能力层
- Skills 解决知识层

MCP 提供：

- 运行代码
- 调用 API
- 读写文件
- 浏览器操作

这些原子能力。

Skills 提供：

- 先做什么
- 后做什么
- 哪些坑要避开
- 哪些规则必须遵守

这些操作经验。

但 Skills 也会冲击一部分“纯知识封装型”的 MCP 服务。如果某个 MCP Server 的核心价值只是告诉 Agent “怎么处理 PDF”，那它很可能更适合被重构成 Skill。

然而，MCP 在企业里仍然有 Skills 无法替代的护城河：

- 权限认证
- 多租户隔离
- 安全边界
- 审计日志
- 外部系统连接基础设施

一句话说：

- Skills 适合封装方法论
- MCP 适合承载能力和安全边界

#### 11.2.10 Skills 的反模式和治理要求

你给出的反模式里，几个点非常值得保留。

第一，大量相似 Skill 并存，没有统一治理。

如果同时存在多个语义重叠的 Skill，例如：

- `parse-pdf`
- `read_pdf`
- `pdf-extract`
- `document-pdf`

主 Agent 的路由会变得高度随机，结果不可复现。

第二，不审查外部 Skill 来源。

来源不明的 `SKILL.md` 和 `scripts/`，本质上和来源不明的 shell 脚本没有本质区别。只要它进入系统，就可能对工具调用链和沙箱执行造成安全风险。

第三，`SKILL.md` 过长不拆分。

如果一个 Skill 说明书写到几百上千行，模型注意力会明显衰减，底部的关键约束可能被忽略。技能说明应该尽量保持克制、聚焦和拆分。

第四，引用链过深。

如果 `SKILL.md` 再引用第二层、第三层文档，很多执行模式根本不会完整读取这些深层引用，结果就是你以为写进去的规则实际上没有生效。

### 11.3 典型反模式

- 同一领域存在多个语义重叠 Skill 且没有统一命名和治理，会让主 Agent 路由随机化、结果不可复现
- 不经审查直接引入外部 `SKILL.md` 或 `scripts/`，等价于运行来源不明的脚本和操作手册
- 把过多场景硬塞进一个超长 `SKILL.md`，会导致模型注意力衰减，关键约束被忽略
- 让 Skill 引用链层层嵌套，超出执行器默认读取范围后，后面写的规则实际上根本不会生效
- 把本该沉淀成稳定脚本的流程全部交给模型临时生成代码，会放大波动性和审核难度

### 11.4 可以沉淀出的最佳实践

- Skills 适合承载“怎么做”的经验，而不是替代所有工具和服务
- Skill 说明书要同时服务决策层和执行层：frontmatter 做路由，正文做执行
- 用渐进式披露管理大规模 Skills 库，避免上下文爆炸
- 主 Agent 负责编排，任务型 Skill 负责在隔离上下文中执行
- Skill 执行应配合最小权限工具白名单
- 能沉淀成稳定脚本的逻辑，优先写进 `scripts/`，不要每次都让模型动态生成
- Skills 和 MCP 应保持知识层与能力层分离
- 企业级 Skill 管理至少要有命名规范、Review 流程、版本管理和使用统计
- 外部 Skill 视同可执行代码，必须做来源审查

## 12. 记忆工程总论：本质、演进与治理框架

当一个 Agent 已经具备工具、沙箱、MCP 和 Skills 之后，下一步真正决定用户体验的，往往不是“它能不能做”，而是“它记不记得”。不记得用户是谁、不记得上午说过什么、不记得已经教过它的工作 SOP，这会让一个原本很强的 Agent 迅速退化成“每次都失忆的工具人”。

这一节最重要的任务，不是直接写代码，而是先把记忆工程的认知框架搭起来：记忆到底是什么、它为什么会演进、又该如何治理。

### 12.1 为什么这样设计

很多人第一次接触 Agent 记忆时，容易误以为“模型本身会记住用户”。但这恰恰是最大的误区。

大模型本身没有记忆。每一次 API 调用都是独立的，模型并不知道上一轮说过什么。所谓“记住”，本质上只是应用层在构造本次请求时，把过去的信息重新塞进了 `message list`。

这意味着：

- 你放进 `messages` 的内容，模型就“知道”
- 你没放进去的内容，模型就“不知道”

所以，记忆工程的本质不是某种神秘能力，而是一套围绕 `message list` 的信息流设计机制。

如果把系统类比成计算机：

- 模型本身像 `CPU`
- 上下文窗口像 `RAM`
- 文件、数据库、向量库等持久化存储像“硬盘”

那么记忆系统真正解决的问题就是：

- 什么时候把硬盘里的信息加载到内存
- 什么时候把内存中的信息写回硬盘
- 当内存快满时，如何主动管理

### 12.2 具体做法

#### 12.2.1 先明确：记忆的本质就是构建 `message list`

从工程视角看，记忆不是模型内部状态，而是请求构造逻辑。

例如：

```python
messages = [
    {"role": "system", "content": "你是 XiaoPaw，一个飞书工作助手..."},
    {"role": "system", "content": "用户偏好：周报格式先汇总再列计划..."},
    {"role": "user", "content": "帮我发周报"},
    {"role": "assistant", "content": "好的，我来按你喜欢的格式..."},
]
```

这里所谓的“记忆”，其实就是第二条 system 消息。它不是模型自己记住的，而是应用层主动塞进去的。

这个认知非常关键，因为它决定了后续所有记忆工程设计都不是“增强模型神秘能力”，而是“精确控制信息流”。

#### 12.2.2 记忆工程经历了三代演进

你给出的演进脉络可以归纳成三代。

第一代是 `Prompt Engineering`。

这个时代的 AI 产品大多只是聊天机器人。上下文基本就等于用户写的一句 prompt，工程师关注的是“怎样措辞能触发模型更好的回答”。这个阶段几乎没有显式的上下文治理问题，因为上下文本来就很短。

第二代是 `Context Engineering`。

当产品开始进入工作流编排、单 Agent 调工具、多步任务执行之后，上下文不再只是用户的话，而会混入：

- 工具返回结果
- 中间步骤结果
- 多 Agent 输出

这时核心问题变成：在有限 token 预算内，如何构建最高质量的上下文。

第三代是 `Harness Engineering`。

当任务复杂到不能提前编排完整步骤时，系统开始从“直接写 prompt”转向“设计环境”。也就是：

- 预加载关键索引
- 预放规则和约束
- 让模型在受控环境中按需探索

这一代的关键变化是：控制权从“写具体内容”转向“造探索环境”。

#### 12.2.3 记忆治理的本质是加法与减法

不管是哪一代，上下文治理最终都可以归结为两件事：

- 加法
- 减法

加法解决的是“模型不知道它该知道的”。

常见做法包括：

- `Bootstrap` 预加载
- 工具返回注入
- 记忆检索注入
- Skill 按需加载

这些都属于往当前上下文里“补信息”。

减法解决的是“上下文里存在不该有的信息”。

常见做法包括：

- 压缩
- 截断
- 选择性遗忘
- Sub-agent 隔离

也就是说，记忆系统不是一味往上下文里加东西，而是必须同时思考“什么该放进来”和“什么必须移出去”。

#### 12.2.4 任何记忆系统都要回答三个问题

把加法和减法再抽象一层，任何记忆系统最终都要回答这三个问题：

- 记什么
- 怎么存
- 怎么取

这三个问题几乎可以看成模块三后续所有内容的总纲。

其中：

- “记什么”决定写入标准
- “怎么存”决定存储结构和生命周期
- “怎么取”决定当前任务是否真的能拿到相关记忆

如果这三件事设计不清楚，记忆系统最终不是“记太少”，就是“记太多”，或者“虽然记了但永远取不对”。

#### 12.2.5 `Context Rot` 是记忆工程必须面对的物理约束

为什么减法不可缺？根本原因就在于 `Context Rot`。

`Context Rot` 不是经验问题，而是 Transformer 架构的物理约束。模型在生成每个 token 时，会对上下文中的所有 token 分配注意力，而注意力预算是有限的。

这意味着：

- 上下文越短，每个 token 分到的注意力越多
- 上下文越长，每个 token 分到的注意力越少

也就是说，模型不是“突然失忆”，而是上下文太大时，注意力被稀释了。

#### 12.2.6 超过 40% 是一个重要警戒线

你给出的一个非常重要的工程结论是：

- 当上下文使用量超过约 `40%` 时，模型能力会开始明显下降

这个数字的工程意义非常直接：

- 不要等上下文快满了才去治理
- 应该在明显逼近上限之前就主动压缩和管理

记忆治理应该是主动机制，而不是快爆窗时的被动补救。

#### 12.2.7 注意力对上下文位置的关注是 U 型分布

除了总长度问题，位置分布也很重要。

研究发现，模型对上下文的注意力通常呈 U 型分布：

- 开头最重要
- 结尾最重要
- 中间最容易被忽略

这能解释很多实践经验：

- System Prompt 放在最前面很重要
- 最近几轮对话最重要
- 中间大段历史最容易信息衰减

这也是为什么像 Claude Code 会在工具调用后持续把关键提醒注入末尾。它其实是在利用 U 型分布的“尾部高注意力区”。

从工程设计角度，这意味着：

- 关键长期规则适合放在开头
- 当前任务关键提醒适合不断顶到末尾
- 中间区域必须精简，不能堆满低价值历史

### 12.3 可以沉淀出的最佳实践

- 把记忆理解为 `message list` 构建机制，而不是模型神秘能力
- 记忆工程首先是信息流设计，其次才是存储方案选择
- 上下文治理必须同时做加法和减法，不能只做其中一边
- 任何记忆系统都应围绕“记什么、怎么存、怎么取”三问设计
- 记忆治理应主动进行，不要等上下文接近上限才补救
- 重要长期规则放在上下文前部，当前关键提醒放在尾部
- 中间区域最容易发生注意力衰减，应优先压缩和清理
- 不要迷信超大上下文窗口，精选信息通常比堆满信息更有效

## 13. 上下文生命周期管理：Bootstrap、剪枝与压缩

在理解了记忆工程的本质之后，下一步真正要解决的问题不是“该不该管理上下文”，而是“在什么时候、通过什么钩子去管理上下文”。对于 Agent 来说，上下文不是像 Workflow 一样由开发者一步步显式写出来的，而是在 ReAct 循环里自动生长出来的。

因此，上下文治理的关键不是直接控制内容，而是识别那些可以动手术的生命周期节点，并在这些节点上做加法或减法。

### 13.1 为什么这样设计

如果不在生命周期节点上主动干预，Agent 的上下文通常会出现三类生产级崩溃。

第一，`Context Overflow`。

随着工具调用、对话轮次和中间结果不断进入 `message list`，上下文会不可避免地触顶。如果没有主动管理机制，最终一定会报 `context_length_exceeded` 之类的错误。

第二，`Context Rot`。

即使还没到窗口上限，长上下文本身也会让模型性能下降。随着噪声增多、历史变厚，模型会开始重复分析、忽略重点、产生矛盾。

第三，`Context Poisoning`。

错误信息一旦进入上下文，就会像污染源一样持续影响后续推理。Transformer 会对整个上下文做注意力分配，并不会自动区分“这条信息是错的所以忽略”。

这三类问题的共同根源是：Agent 的上下文会自己生长，但不会自己治理。

### 13.2 具体做法

#### 13.2.1 先明确：Agent 的上下文是“生长”出来的

Workflow 和 Agent 最大的区别之一，就在于上下文控制权。

在 Workflow 里，开发者显式决定：

- Step 1 输出什么
- Step 2 接收什么
- 哪些中间结果继续往下传

而在 Agent 的 ReAct 循环里，模型会自己决定：

- 什么时候思考
- 什么时候调工具
- 调什么工具
- 工具结果如何回流进上下文

因此，Agent 上下文并不是你提前写好的，而是运行过程中生长出来的。开发者能做的，是在特定节点上拦截和调整。

#### 13.2.2 三个最关键的操作：Bootstrap、剪枝、压缩

围绕生命周期管理，这一节的三个核心动作分别是：

- `Bootstrap`
- `Pruning`
- `Compaction`

其中：

- `Bootstrap` 是典型加法
- `Pruning` 和 `Compaction` 是典型减法

它们分别解决不同问题：

- `Bootstrap` 解决“启动时模型不知道该知道什么”
- `Pruning` 解决“工具结果膨胀太快”
- `Compaction` 解决“历史对话越积越厚”

#### 13.2.3 `Bootstrap`：给 Agent 一个可靠起点

`Bootstrap` 的目标，是在 Agent 正式开始任务前，把它必须知道的内容主动注入上下文。

这类信息通常分两类：

- 身份与规则
- 状态与记忆

你给出的设计里，`build_bootstrap_prompt()` 读取：

- `soul.md`
- `user.md`
- `agent.md`
- `memory.md`

然后分别包裹成：

- `<soul>`
- `<user_profile>`
- `<agent_rules>`
- `<memory_index>`

这种结构化标签有两个好处：

- 让模型更容易定位不同信息块
- 让每类信息都能单独治理和截断

这里还有一个很关键的原则：Bootstrap 只加载“导航骨架”，而不是把所有文件全文塞进去。尤其是 `memory.md`，只取前 200 行，就是为了把它当成“记忆索引”而不是“完整记忆仓库”。

#### 13.2.4 `Pruning`：最先控制的是 Tool Result 膨胀

实际运行里，最大的上下文膨胀源通常不是用户对话，而是工具返回结果。

一次搜索工具、网页抓取工具或文件读取工具，就可能带回成千上万 token。如果不加控制，十几次工具调用之后，上下文会迅速被原始工具输出吞噬。

你给出的 `prune_tool_results()` 策略非常典型：

- 找到倒数第 `N` 个 user 消息
- 在此之前的 `tool` 消息不删除结构，只把内容替换为 `[已剪枝]`

这个设计的关键点是：

- 释放 token
- 保留消息结构
- 保留 `tool_call_id` 等引用关系

也就是说，剪枝不是“粗暴删除”，而是“保留框架依赖的骨架，去掉模型已经不需要的原材料”。

#### 13.2.5 `Compaction`：把旧历史变成摘要

当仅靠剪枝还不够时，就需要压缩。

压缩的基本策略是：

- 保留 system 消息
- 保留最近若干轮“新鲜区”
- 把更早的历史分块摘要
- 用结构化摘要替换旧消息

你给出的 `maybe_compress()` 实现里，关键设计包括：

- 使用上下文比例阈值触发，而不是等彻底超限
- 按 user 消息边界切割，而不是按消息条数切
- 使用轻量模型做摘要，而不是拿大模型做低创造性任务

尤其是“以 user 消息边界切割”这一点非常重要。因为 assistant 的 `tool_call` 和 tool result 必须成对保留，如果硬按条数切，很容易切断这类配对关系，直接把框架打崩。

#### 13.2.6 在 CrewAI 里，`backstory` 和 `@before_llm_call` 是两个关键干预点

你给出的 CrewAI 落点非常清楚。

第一个干预点是 `backstory`。

这决定了 Bootstrap 注入发生在 Agent 定义阶段。也就是说，Agent 的 system prompt 不是写死的一段字符串，而是由 `build_bootstrap_prompt()` 动态构造的。

第二个干预点是 `@before_llm_call`。

这是在每次模型调用前，拦截并修改 `context.messages` 的机会。它可以同时承担：

- session 恢复
- 剪枝
- 压缩

这也是为什么 `before_llm_hook()` 成了整套上下文生命周期管理的核心入口。

#### 13.2.7 `@before_llm_call` 的关键约束是 in-place 修改

这里有一个非常重要的框架细节：`context.messages` 必须做 in-place 修改。

正确做法是：

- 改单条消息内容
- `clear()`
- `extend()`

错误做法是：

- 直接写 `context.messages = [...]`

原因在于，`context.messages` 实际上是 executor 内部列表的直接引用。重新赋值只会让 Python 变量名指向一个新列表，但 executor 仍然持有旧引用，框架完全感知不到你的修改。

这一点属于非常典型的“框架接线细节”，如果忽略它，表面上代码跑了，实际上治理根本没生效。

#### 13.2.8 Session 恢复和压缩是两件独立的事

这套实现里还有一个容易混淆的点：session 恢复和压缩快照虽然都作用于上下文，但目标不同。

session 恢复的目标是：

- 让 Agent 看到连续的历史

它通过读取 `{session_id}_ctx.json`，把历史 `messages` 恢复回来，再把当前 user 消息追加进去。

而压缩的目标是：

- 控制上下文长度和质量

它是在 session 继续推进过程中，根据阈值动态执行。

因此：

- `ctx` 快照负责“恢复”
- `raw` 历史负责“lossless 保底”
- `before_llm_call` 负责“持续治理”

这三者是互补关系，而不是同一件事。

#### 13.2.9 `raw` 和 `ctx` 双写是一种 lossless 策略

你给出的持久化方案非常值得保留：

- `ctx.json`：压缩过的恢复快照
- `raw.jsonl`：append-only 的完整原始历史

这里的核心思想是：

- 压缩是有损的，这是设计，不是 bug
- 所以必须保留一份 lossless 原始日志

这样即使压缩过程丢掉了细节：

- 运行恢复仍然能靠 `ctx`
- 调试、审计、复盘仍然能靠 `raw`

这比只保存一个被压缩后的快照稳健得多。

#### 13.2.10 这套体系背后依赖的是 Hook 作用域和执行时机

从底层看，`@before_llm_call` 在 CrewAI 里并不是随便一个装饰器，而是：

- 先给方法打标记
- 再由 `@CrewBase` 初始化时自动扫描并注册
- 绑定成当前 Crew 实例的 bound method
- 在每次 LLM 调用前统一触发

这意味着两个结论：

- Hook 是 Crew 级别作用域，不会天然污染其他 Crew
- 你可以利用 `context.agent`、`context.task`、`context.iterations`、`context.llm` 等信息做更细粒度治理

也就是说，一旦理解了这个执行时机，你不仅能做剪枝和压缩，还能进一步扩展出：

- 迭代上限控制
- 多 Agent 差异化策略
- 动态阈值治理

### 13.3 典型反模式

- 误把 `memory=True` 当成完整上下文管理方案，会掩盖 Bootstrap、压缩和生命周期治理的缺失
- 等上下文真正超限再处理，通常为时已晚，模型在此之前就已经发生 `Context Rot`
- 在 Hook 里直接重赋值 `context.messages = [...]`，修改对框架实际上不会生效
- 压缩时不保护 `tool_call` / `tool_result` 配对关系，会直接打断引用链并导致执行崩溃
- 压缩粒度过细、逐条消息摘要，会丢掉回合级语义，还会制造大量低价值摘要调用
- 写 `ctx` 快照或会话文件时不做原子替换或并发保护，在多请求场景下很容易损坏 session 文件

### 13.4 可以沉淀出的最佳实践

- Bootstrap、剪枝、压缩分别对应启动前注入、运行中减负、历史摘要化
- Bootstrap 应只加载导航骨架，不要把所有内容全文塞进 system prompt
- Tool Result 往往是最先需要治理的膨胀源
- 剪枝时优先保留消息结构和引用关系，不要直接破坏 tool call 配对
- 压缩应在较早阈值触发，不要等上下文逼近上限才处理
- 压缩切割应按 user 边界或完整回合边界进行，避免打断执行链
- CrewAI Hook 修改 `context.messages` 时必须 in-place 操作
- Session 恢复、上下文压缩和原始历史持久化应拆成独立职责
- 对有损压缩，必须配套 lossless 原始日志保底
- 摘要任务优先用小模型，对话任务优先用大模型，以控制成本

## 14. 知识记忆写入与治理：`memory-save`、`skill-creator` 与 GC 闭环

如果说前一节解决的是“上下文如何在一次运行里被管理”，这一节解决的就是“哪些知识值得跨 session 留下来，以及如何把它们变成稳定、可治理、可复用的长期资产”。

这里最重要的，不是简单地“把内容写到文件里”，而是建立三类机制：

- `memory-save`：有控制地写入知识记忆
- `skill-creator`：把可复用 SOP 沉淀成程序记忆
- `memory-governance`：定期做垃圾回收和治理

### 14.1 为什么这样设计

长期记忆如果没有写入标准和治理标准，通常很快会走向两种失败：

- 记得越来越多，但越来越乱
- 写入本身很容易，治理成本越来越高

更进一步，长期记忆里混入错误信息、重复偏好、过时事实、无边界 SOP 或跨用户污染后，它的破坏力会比“完全没有记忆”更大。因为 Agent 会把这些东西当成事实，在未来每次 Bootstrap 时重新加载，错误会被反复放大。

所以，这一节的核心不是“怎么多存点”，而是：

- 什么信息该进持久化记忆
- 该进普通 memory，还是该升格为 Skill
- 记忆写入之后如何形成自动治理闭环

### 14.2 具体做法

#### 14.2.1 `memory-save` 的触发要分显式与主动两类

`memory-save` 的第一个工程问题，不是写到哪，而是何时触发。

两种主要方式是：

- 用户显式触发
- Agent 主动触发

显式触发最可靠，因为用户直接表达了“这件事下次还要用到”，信号清晰，误判率低。

主动触发则依赖 `memory-save` 的 `SKILL.md` description 里预先写清触发场景，例如：

- 用户描述自己的偏好
- 用户暴露长期工作习惯
- 用户表达稳定的协作方式

但实践里要注意，模型的主动记忆意识通常不强，所以这类场景必须写得足够具体，而且更适合发生在一个自然结束点，而不是对话进行到一半时突然打断去记。

#### 14.2.2 `memory-save` 更适合做成 Skill，而不是直接做成 Tool

直觉上，完全可以写一个 `MemorySaveTool(BaseTool)` 来处理记忆写入。但你给出的两个限制很关键。

第一，写入规范太丰富，普通 `description` 装不下。

`memory-save` 需要明确：

- 什么时候触发
- 什么时候不触发
- 四种写入目标各自格式
- 四信号准入控制
- `memory.md` 的行数治理规则

这些内容更适合放在 `SKILL.md` 里，用结构化正文、规则块和示例展开，而不是挤进一段工具 description。

第二，Skill 更容易配合安全隔离。

主 Agent 不应直接持有全局写文件能力。正确做法是：

- 文件写入封装进 Sub-Crew
- Sub-Crew 通过沙盒 MCP 操作文件
- 沙盒只挂载指定目录

这样“写到哪”由系统挂载决定，而不是靠 prompt 里一句“只能写这里”。

#### 14.2.3 长期记忆文件结构要分清角色：索引、偏好、规则、主题

你给出的 `workspace/` 结构里，关键文件职责非常清楚：

- `soul.md`：固定人设，不由 `memory-save` 修改
- `user.md`：用户偏好和习惯
- `agent.md`：Agent 行为规范的增量自我进化
- `memory.md`：记忆索引，只存指针，不存正文
- `memory_{topic}.md`：按主题拆开的详细记忆文件

这里最核心的是 `memory.md`。

它不是正文仓库，而是目录层。也就是说：

- Bootstrap 全量加载的是目录
- 详细内容按需去 topic 文件读取

这样做的好处是：

- `memory.md` 可以保持很薄
- topic 文件可以很厚
- Bootstrap 成本稳定
- 真正需要时再渐进式披露

#### 14.2.4 `memory-save` 的核心是四步规范

你给出的四步规范可以看成整套写入逻辑的标准路径。

第一步，准入控制。

不是所有信息都值得进入持久化记忆。写入前先过四个信号：

- `Utility`
- `Confidence`
- `Novelty`
- `Type`

也就是：

- 三个月后还有没有价值
- 当前内容是否有直接证据支撑
- 目标文件里是否已有同类内容
- 它到底是稳定偏好、规则、决策，还是瞬态信息

这一步最好在主 Agent 决定调用 `memory-save` 之前就完成，因为写入成本很低，治理成本很高。

第二步，分类路由。

不同类型信息写到不同位置，例如：

- `user` → `user.md`
- `agent` → `agent.md`
- `memory_index` → `memory.md`
- `topic` → `memory_{name}.md`

这一步本质上是在决定“这条知识属于哪种记忆层级”。

第三步，写入前检查。

重点是 `memory.md` 的行数门控：

- `< 150` 行：正常写入
- `150–179` 行：预警，附带建议触发治理
- `>= 180` 行：拒绝写入，要求先治理

这里的重要性在于：治理不是等 200 行才想起来，而是提前留缓冲。

第四步，更新优于追加。

写入前先读目标文件，查找是否已有相关旧记忆：

- 找到旧记忆 → `str_replace` 更新
- 找不到 → 追加新条目

这可以避免出现多条相互冲突的旧版规则并存。

#### 14.2.5 每条记忆都应该带双时间戳

你给出的标准条目格式非常重要：

```text
- 代码注释用中文  [created: 2026-01-15, updated: 2026-03-10]
```

`created` 表示首次产生时间，`updated` 表示最近确认或修改时间。

这两个字段的价值不只是“好看”，而是为后续治理提供依据：

- 哪些信息长期没被确认
- 哪些信息可能过期
- 哪些主题最近仍活跃

没有时间戳，后续 GC 基本只能凭感觉。

#### 14.2.6 不是所有“记下来”的东西都该进 memory，有些应该升级成 Skill

这里的关键区分是：

- 语义记忆（Semantic Memory）
- 程序记忆（Procedural Memory）

像“用户喜欢短回复”“用户偏好中文注释”这种，属于语义记忆，用 `memory-save` 存很合适。

但像：

- 港股分析 SOP
- 客户投诉处理流程
- 代码 review 七步检查清单

这类多步骤流程，即使写进 `memory.md` 或 `memory_{topic}.md`，模型下次执行时也未必稳定，会漏步骤、改顺序、忽略关键约束。

这时更适合把它沉淀成 Skill。因为 Skill 具备：

- frontmatter 路由
- 正文步骤说明
- `CRITICAL` 规则
- 可复用和可迁移的执行结构

一句话说：

- 记“是什么 / 有什么” → `memory-save`
- 记“怎么做 / 必须稳定执行” → `skill-creator`

#### 14.2.7 `skill-creator` 负责把 SOP 沉淀成程序记忆

你给出的简化版流水线非常适合作为工程落地骨架：

- `Capture Intent`
- `Draft SKILL.md`
- `Register`

其中：

第一步 `Capture Intent` 负责从用户对话中提炼：

- Skill 名称
- 触发场景
- 执行步骤
- 最关键的 `CRITICAL` 约束

第二步 `Draft SKILL.md` 负责生成完整 Skill 文件，包括：

- frontmatter
- pushy 的 description
- explain the why 的正文步骤

第三步 `Register` 负责：

- 检查是否同名
- 写入 `/mnt/skills/{skill-name}/SKILL.md`
- 更新 `load_skills.yaml`
- read-back 验证

这样一来，新 Skill 在下次 session 启动时就会被 `SkillLoaderTool` 自动识别，无需改代码。

#### 14.2.8 `memory-governance` 是记忆系统的 GC

长期记忆如果只写不治理，最终一定会腐化。你给出的三种腐化形态非常关键：

- 过期事实
- 相互矛盾
- 死链 / 野文档

所以记忆系统必须有自己的 GC，也就是 `memory-governance`。

它的触发方式可以有三种：

- 阈值触发
- 定期触发
- 用户主动触发

其中最好的做法，是让它尽量自动发生，而不是依赖人想起来。

#### 14.2.9 `memory-governance` 应先扫描、报告，再等待确认后治理

你给出的治理流程是：

- 扫描
- 分析
- 生成报告
- 等待用户确认
- 再执行清理

这个顺序非常重要。尤其是在涉及删除、归档、禁用 Skill 之类操作时，不能让 Agent 在没有确认前直接动手。

扫描项可以包括：

- 行数和死链
- 野文档
- 路由错配
- 表述冲突
- 表述冗余
- Skills 健康度
- 安全扫描
- 过期条目

这本质上已经不只是“清记忆”，而是在做一套知识仓库审计。

#### 14.2.10 文件记忆的底层哲学仍然是渐进式披露

这一节里 `memory.md` 的设计，本质上延续了 Skills 里同样的设计哲学。

也就是：

- 目录层很薄
- 内容层按需读取

区别只在于：

- Skills 的渐进式披露更多是代码驱动的
- memory 文件体系的渐进式披露更多是模型自驱的

也就是说，Agent 先读 `memory.md` 看到：

- 某个主题在哪个文件

再决定需不需要读取对应 topic 文件。

这样一来，Bootstrap 始终轻量，真正详细的主题内容只有在相关任务里才进入上下文。

### 14.3 典型反模式

- 把外部工具原始输出或来源不明的内容直接写进长期记忆，会把 Prompt Injection 或幻觉结果持久化，并在后续 session 中持续放大
- 把所有对话内容都当成值得保存的记忆，会让记忆系统迅速变成噪声仓库，最终比没有记忆更难用
- 只追加、不更新，会让同一主题出现多条互相冲突或过时的记录，模型无法判断哪条才是最新状态
- 把多步骤 SOP 直接写进普通 memory 文件，而不是升级成 Skill，会导致模型“知道有这回事”但执行时仍然漏步骤、换顺序、忽略约束
- 长期只写不治理，会让过期事实、死链、野文档和冲突条目不断堆积，记忆系统逐渐腐化
- 在多用户场景下共用同一套 workspace 和记忆文件，会直接引发跨用户信息泄露和合规风险
- 治理动作不经过报告和确认就直接删除或修改记忆文件，容易误删长期有效知识，而且损失不可逆

### 14.4 可以沉淀出的最佳实践

- 记忆写入首先解决“准不准写”，其次才是“怎么写”
- `memory-save` 更适合承载复杂写入规范和安全边界，不应只是一个简单写文件工具
- `memory.md` 只做索引，不做正文仓库
- 每条长期记忆应带 `created` / `updated` 双时间戳
- 更新优于追加，长期记忆应越来越准确，而不是越来越长
- 偏好、事实、稳定信息适合进 memory；多步骤 SOP 更适合升级为 Skill
- `skill-creator` 适合把可重复工作方法沉淀为程序记忆
- 记忆系统必须内建 GC，而不是长期只写不管
- 治理前先生成报告并等待确认，避免误删长期有效知识
- 长期记忆体系也应遵守渐进式披露：索引常驻，正文按需读取
