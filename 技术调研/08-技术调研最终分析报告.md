# Super-Agent 技术调研

> 汇报主题：Super-Agent 前期技术调研、框架选型、自研边界与实施计划  
> 推荐受众：技术负责人、部门负责人、项目决策人  
> 汇报目标：从技术角度说明下一代企业智能体的框架调研结论、能力复用边界、工程风险和阶段性建设计划。
> 调研边界：本文只讨论下一代企业智能体的技术实现路径和工程组合方式，不涉及对既有产品线、既有 Agent 能力或其他团队方案的业务评价。

---

## 1. 调研背景与目标

我们的目标是构建下一代企业智能体，面向企业内部复杂知识工作、工具协同和业务执行场景，形成可持续演进、可维护、可治理的智能体能力体系。

核心判断维度包括：Agent 编排能力如何复用成熟技术，模型、工具、Skill、MCP、沙箱、安全审计和可观测能力如何形成统一标准，以及哪些控制面能力需要结合企业场景做工程化建设。

整体思路是：成熟能力优先复用，关键控制面按企业场景沉淀，避免重复建设，也避免过度耦合。

### 现有产品线技术架构背景

从前期背景调研看，当前各产品线已经在不同业务场景中形成了各自的 Agent 技术实现路径。

| 产品线 / 场景 | 当前技术架构 | 技术特征 |
|---------------|--------------|----------|
| R&D | 基于 `pydantic-ai` 自研 | 类型体系清晰，适合在 Python 工程内做轻量 Agent 能力沉淀 |
| ip-drafting | `LangGraph` + `LangChain`，手动实现编排 | 图编排能力强，适合流程明确、状态流转复杂的任务 |
| ls | `CrewAI` | 角色和任务抽象清晰，适合固定流程、角色边界明确的业务 |
| analytics | `Claude Code SDK` | 工具链完整，适合代码、分析和项目上下文较重的场景 |
| mse | `LangChain` + `LangGraph` | 生态完整，适合基于 LangChain 体系构建复杂链路 |
| ip-checking | `LangGraph` + OpenAI SDK，手动实现 agent loop | 灵活度高，适合对执行循环和模型调用有定制要求的场景 |

企业内部 Agent 技术路线已经呈现多框架、多实现、多运行时并存的状态。不同产品线根据自身业务诉求做了合理取舍，但从下一代企业智能体的技术建设角度看，也带来几个共性问题：

- Agent 编排层实现方式不统一，后续能力复用和跨场景迁移成本较高。
- 多模型接入、工具调用、MCP、Skill、沙箱、安全审计等能力容易在不同产品线重复建设。
- 手动实现 agent loop 的方案灵活度高，但长期需要承担上下文管理、工具执行、错误恢复、观测和安全治理成本。
- LangChain / LangGraph / CrewAI / Claude Code SDK 等方案各有优势，但运行时假设、依赖体系和扩展方式不同，需要在企业级平台层做统一抽象。

因此，本次 Super-Agent 技术路线会在充分理解现有技术分布的基础上，沉淀一套更适合长期维护、统一治理和跨场景复用的企业智能体工程架构。

从这个背景出发，后续技术方案需要突出几个特点：

- 编排内核要轻量，避免把业务系统过度绑定到单一大型框架。
- 多模型、工具、Skill、MCP、沙箱和安全策略要有统一治理层。
- 运行时能力要可观测、可审计、可回放，满足企业级维护要求。
- 通用能力优先复用，差异化控制面在 Super-Agent 工程体系内沉淀。

## 2. 候选方案范围

本次调研覆盖三类对象：

- SDK 型框架：`pydantic-deep`(pydantic-ai)、Deep Agents(langchain)、CrewAI。
- 应用型或平台型框架：DeerFlow、AgentScope、Hermes。
- 沙箱执行引擎：Pi-mono CLI、Claude API 直调、Claude Code CLI。

SDK 型框架更适合作为编排组件集成到工程体系中；应用型或平台型框架更适合作为工程设计、平台能力和安全治理参考；沙箱执行引擎单独评估，因为它直接影响代码、脚本、Skill 和工具执行的安全边界。

### 框架定位对比

| 方案 | 类型 | 技术特长 | 关注点 | 在 Super-Agent 中的定位 |
|------|------|--------|----------|--------------------------|
| pydantic-deep | SDK 型 | 轻量、可嵌入、依赖少、task 委派自然 | pre-1.0，部分能力需补强 | 作为当前编排内核候选 |
| Deep Agents | SDK 型 | 沙箱抽象、权限模型、威胁建模 | API 波动大，LangChain 依赖重 | 作为执行抽象参考 |
| CrewAI | SDK 型 | 显式流程、商业化成熟 | 更偏固定流程编排，角色调优成本较高 | 固定流程场景参考 |
| DeerFlow | 应用型 | 中间件、安全审计、错误处理 | LangChain/LangGraph 锁定重 | 工程质量参考 |
| AgentScope | 平台型 | 多 Agent、记忆、状态、观测 | 无内置沙箱，核心实现偏重 | 多 Agent 能力蓝图 |
| Hermes | 应用型 | 多通道、工具接入 | 自研逻辑重，通用 SDK 属性弱 | 局部能力参考 |

> 我们没有只看一个框架，而是把 Agent SDK、完整 Agent 应用、自研架构和沙箱执行引擎都纳入了评估。整体判断是：SDK 型框架适合承担编排组件，应用型框架适合提供设计参考，沙箱执行引擎需要单独评估。

## 3. 选型判断标准

选型重点从“框架功能完整度”转向“工程适配度”。Super-Agent 已经具备请求入口、模型路由、主控编排层、上下文装配、工具注入、Sub-Agent 委派、事件流输出和安全治理等控制面，因此 Agent SDK 更适合作为编排能力组件，而不是主导整个系统结构。

核心判断标准如下：

### 判断标准表

| 判断维度 | 关注问题 | 对 Super-Agent 的重要性 |
|----------|----------|--------------------------|
| 架构契合度 | 是否匹配动态路由和 Sub-Agent 委派 | 最高 |
| 依赖风险 | 是否深度绑定 LangChain 等重依赖 | 高 |
| 可演进性 | 是否能降级、扩展或 fork | 高 |
| 工程可控性 | 团队能否自行修复和维护 | 高 |
| 安全治理 | 权限、审计、沙箱是否可控 | 最高 |
| 扩展能力 | 是否支持后续平台化演进 | 中高 |

> 我们这次不是按功能数量打分，而是按“是否适合当前工程组合”来判断。因为 Super-Agent 已经有自己的控制面，编排组件需要轻、稳、可集成、可演进，并与上层架构保持清晰边界。

## 4. 技术路线建议：采用 pydantic-deep 作为编排内核

当前工程组合建议以 `pydantic-deep v0.3.3` 承担 Agent 编排内核能力。它的核心价值在于轻量、低耦合，以及与 `pydantic-ai` 生态的天然衔接。

Super-Agent 的目标不是让开发者提前写死一组 Agent、Task 和执行流程，而是由系统根据用户请求和上下文动态判断执行模式。例如 DIRECT、AUTO、PLAN_AND_EXECUTE、SUB_AGENT 等模式都需要在运行时决策。这种设计更适合“控制面做模式判断，Agent loop 做执行推进”的分层方式。

`pydantic-deep` 的 `task()` 委派模式适合这个分层：上层主控编排层决定是否需要进入复杂任务模式，编排内核在执行过程中可以根据需要委派子任务、调用工具或收敛结果。从技术特征看，它更偏轻量嵌入；Deep Agents 的执行抽象更完整；CrewAI 的显式流程编排更适合固定流程；DeerFlow 和 AgentScope 则更适合作为完整工程范式参考。

### 核心优势

| 优势 | 说明 |
|------|------|
| 编排模型契合 | LLM 自主决策 + `task()` 委派，与主控编排分层天然匹配 |
| 依赖链轻 | 主要依赖 `pydantic-ai`，避免深度绑定 LangChain/LangGraph |
| 可降级 | 必要时可以退回原生 `pydantic-ai` |
| 可维护 | 源码规模较小，团队可阅读、patch 或 fork |
| 集成成本低 | 对现有架构侵入较小，可以与当前控制面协同 |

### 风险承认

`pydantic-deep` 也存在工程风险。它目前仍是 `v0.3.3`，属于 pre-1.0 阶段，API 稳定性需要通过锁版本和 staging 验证来控制。部分工程能力，例如安全审计、异常分级、沙箱策略和可观测，需要在 Super-Agent 自己的架构层补齐。

但这些风险是可控的，因为它的依赖链较轻、源码规模可控，并且我们保留了降级到原生 `pydantic-ai` 的路径。

> 当前建议以 `pydantic-deep` 承担 Agent 编排内核能力。主要考虑是它轻、可嵌入、依赖少、可降级，和我们当前的主控编排、Sub-Agent、工具注入和动态路由设计契合度较高。

## 5. 其他框架的技术定位与借鉴价值


- DeerFlow：工程质量、安全审计和中间件设计成熟，适合作为工程治理参考。
- AgentScope：多 Agent、记忆系统和观测能力完整，适合作为平台能力蓝图参考。
- Deep Agents：执行抽象、权限模型和威胁建模清晰，适合作为沙箱与权限设计参考。
- CrewAI：显式流程、角色化编排和商业化成熟度较高，适合固定流程任务参考。
- Hermes：多通道和工具接入有局部亮点，适合作为工具生态接入参考。


#### DeerFlow

DeerFlow 的优势非常明显，尤其是在中间件体系、安全审计和错误处理上。它的有序中间件、工具错误处理、循环检测和沙箱审计都值得 Super-Agent 参考。

从集成角度看，DeerFlow 重度依赖 LangChain/LangGraph，整体更像一个完整 Agent 应用，而不是轻量 SDK。如果作为组件集成，需要评估它的图结构、运行时假设和工程组织方式带来的维护成本。

因此 DeerFlow 的定位更适合作为工程质量和安全设计参考。

#### AgentScope

AgentScope 在多 Agent 协作、状态管理、记忆系统和可观测性方面非常成熟。它的 MsgHub、状态序列化、分层记忆和 OTel 观测都具有参考价值。

从集成角度看，它没有内置沙箱能力，命令执行和安全审计需要额外补齐。同时它的核心实现相对偏重，需要结合现有控制面边界谨慎拆解。

因此 AgentScope 更适合作为未来多 Agent 平台能力蓝图。

#### Deep Agents

Deep Agents 的优点是抽象边界清晰，尤其是 BackendProtocol、PermissionsMiddleware、SummarizationMiddleware 和威胁建模意识。这些能力对 Super-Agent 很有价值。

从工程落地角度看，它的 API 变化较快，版本稳定性需要观察，同时依赖 LangChain 生态，升级风险和长期维护成本需要评估。

因此 Deep Agents 更适合作为执行后端抽象和权限模型的设计参考。

#### CrewAI

CrewAI 的优势是流程化、角色化和商业化成熟，适合固定流程任务。例如固定研究流程、固定内容生产流程或固定审批流程。

Super-Agent 的目标是动态路由和动态委派。我们希望系统根据请求实时判断是否需要规划、是否需要工具、是否需要子代理。CrewAI 的显式编排方式更适合流程稳定、角色明确的任务场景。

此外，前期使用中已经发现 CrewAI 的角色、backstory、manager 行为需要较多调优，集成成本偏高。

#### Hermes

Hermes 在多通道接入和工具集成上有一些局部亮点，整体工程假设较强。对于当前 Super-Agent 来说，它更适合作为局部能力参考。

### 对比结论

| 框架 | 技术关注点 | 可借鉴价值 |
|------|------------------|----------|
| DeerFlow | 应用型框架，LangChain/LangGraph 依赖较重 | 安全审计和中间件 |
| AgentScope | 平台能力完整，沙箱能力需额外补齐 | 多 Agent、记忆和观测 |
| Deep Agents | API 变化较快，LangChain 依赖较重 | 执行抽象和权限模型 |
| CrewAI | 显式编排，更适合固定流程 | 固定流程任务参考 |
| Hermes | 工程假设较强，通用 SDK 属性较弱 | 工具和多通道接入参考 |

> 其他框架都有各自的技术价值。本次调研的结论是把不同框架的优势放到合适的位置：编排内核采用轻量可控的 `pydantic-deep`，工程治理参考 DeerFlow，多 Agent 和记忆参考 AgentScope，执行抽象和权限模型参考 Deep Agents。

## 6. 沙箱执行引擎选型

沙箱执行引擎单独评估，不直接跟随 Agent SDK。Super-Agent 的高风险执行场景包括代码生成、文件操作、脚本执行、Skill 执行、工具调用和结果文件回收，这些能力必须通过沙箱执行引擎隔离。

综合功能完整度、成本、工程投入和现有集成情况，当前建议继续以 Pi-mono 作为沙箱主链路；Claude API 直调作为轻量补充；Claude Code CLI 保留为能力参考。

### 核心对比

| 方案 | 优点 | 问题 | 结论 |
|------|------|------|------|
| Pi-mono CLI | 完整 agent loop、并行工具、session、hooks、compaction、多模型 | Node.js 子进程启动有一定开销，JSONL 解析需适配 | 继续作为主方案 |
| Claude API 直调 | 进程内调用，最小上下文，轻量可控 | 需要自建 loop、streaming、并行工具、hooks、session、压缩 | 作为补充方案 |
| Claude Code CLI | 项目感知和工具链完整 | 上下文开销较高，成本和时延需要控制 | 保留为能力参考 |

测试结论是：Pi-mono 与 Claude API 直调整体性能接近，但 Pi-mono 已经具备完整 agent loop、并行工具、上下文压缩、session、hooks 和多模型协议支持。如果用 Claude API 直调补齐这些能力，需要额外建设周期。Claude Code CLI 功能完整，但上下文和成本开销较高。

## 7. 自研与复用边界

总体原则是：通用能力优先复用，平台控制面做工程化建设。Agent loop、模型调用、基础工具包装、MCP 连接等能力已有成熟生态；网关、状态、编排、能力接入、安全治理、观测和前端交互能力则需要结合企业场景统一设计。

### 边界划分

| 能力模块 | 建议方式 | 原因 |
|----------|----------|------|
| 基础工程与配置底座 | 工程化沉淀 | 需要统一项目结构、配置管理、日志规范、构建和部署约束 |
| API 网关与会话入口 | 工程化沉淀 | 需要统一请求入口、会话管理、取消控制和管理接口 |
| 流式协议与事件通道 | 工程化沉淀 | 需要统一 SSE / WebSocket / A2UI 事件协议和恢复机制 |
| 会话状态与上下文持久化 | 工程化沉淀 | 需要管理 Session 生命周期、历史裁剪、TTL 和恢复 |
| Agent loop | 复用 `pydantic-deep` | 降低自研成本，保持轻量嵌入 |
| 模型基础调用 | 复用 `pydantic-ai` | 生态稳定，类型友好 |
| Prompt / Context 装配 | 工程化沉淀 | 需要按模式、角色、用户上下文和记忆统一装配输入上下文 |
| 主控编排 / 调度层 | 工程化沉淀 | 关系到业务模式路由和产品差异化 |
| MCP 基础连接 | 复用 `fastmcp` / `pydantic-ai` | 避免重复建设协议层 |
| Worker / Toolset 组织层 | 工程化沉淀 | 需要统一工具包装、结果转换、事件推送和能力编排 |
| 沙箱内执行 | 复用 Pi-mono | 功能完整，成本可控 |
| 多模型路由 | 自研薄层 | 需要适配内部模型、网关和角色策略 |
| Sub-Agent 协作 | 工程化沉淀 | 需要管理委派协议、并发限制、生命周期和结果聚合 |
| Skill 系统 | 工程化沉淀 | 涉及知识模板、脚本执行和沙箱分流 |
| 记忆系统 | 工程化沉淀 | 涉及用户事实、会话压缩、检索增强和历史复用 |
| 安全权限 | 自研或强控制 | 不能完全交给模型或第三方框架 |
| 事件流协议 | 工程化沉淀 | 直接影响前端 A2UI 和用户体验 |
| 可观测与计费 | 自研集成 | 关系到企业化交付和运营 |
| 前端 A2UI 渲染与业务组件 | 工程化沉淀 | 需要把思考、步骤、工具结果、子代理状态和产物统一可视化 |
| 测试体系与质量保障 | 工程化沉淀 | 需要覆盖接口、流式链路、Agent / Worker 集成和回归测试 |
| 部署、运维与国际化 | 工程化沉淀 | 关系到环境落地、排障能力和多语言交付 |

这里的关键边界是：模型输出不是授权，所有工具执行都要经过系统策略层；权限、安全、审计和沙箱策略必须由系统独立校验。

## 8. 运行时组件设计决策

运行时架构不是单个 Agent SDK 能独立解决的问题，它涉及模型选择、上下文装配、工具注入、沙箱触发、MCP 接入、Skill 执行、事件推送、审计和观测。更合理的整理方式不是按代码目录拆，而是按五个技术层次归纳：

1. 执行内核：负责沙箱内任务执行和代码/脚本类高风险动作。
2. 模型体系：负责多模型配置、Provider 差异收口、角色模型绑定和运行时策略。
3. 行为治理：负责 Hook、Capability、Toolset 的职责划分和工具调用治理。
4. 能力接入：负责 Base Tools、Skill、MCP、Sandbox 等能力统一进入 Agent 可调用范围。
5. 运行时策略：负责把执行模式、模型能力、工具权限、安全审计和事件输出集中收口。

核心价值是避免运行时逻辑散落在 API 层、Agent 工厂、工具函数和 Worker 中，降低后续维护成本。

### 多模型架构决策

多模型问题的本质不是“如何新增一个模型”，而是“如何让业务层不感知模型提供方差异”。

当前如果继续通过模型名字符串判断 provider，例如通过模型名包含 `claude` 判断是否走 Anthropic 原生路径，后续会越来越难维护。更合理的方向是引入分层多模型架构：

| 层次 | 职责 |
|------|------|
| Catalog 层 | 管理 provider、model profile、role binding，配置以 `models.yaml` 为主 |
| Provider 适配层 | 把模型配置转换为真正可执行的模型对象，隔离不同 provider 差异 |
| Registry / Facade 层 | 对外暴露 role-based 查询入口，例如按角色和执行模式获取模型 |
| Runtime Policy 层 | 根据 execution mode 与 model capability 生成运行时策略 |
| Compatibility 层 | 统一处理 reasoning、tool calling、streaming、transport 等兼容差异 |

最终目标是让业务层只依赖逻辑角色，不直接依赖 provider 细节。模型能力差异进入 capability / compatibility 建模，运行时策略由 runtime policy 统一收口。

### Hook、Capability、Toolset 职责划分

| 组件 | 职责定位 | 适合承载的能力 |
|------|----------|----------------|
| Hook | 轻量观察、拦截、审计 | 事件记录、工具调用前后拦截、轻量校验 |
| Capability | 深层行为注入 | 状态隔离、Prompt 动态控制、工具动态控制、复杂上下文注入 |
| Toolset | 工具组织、包装、过滤和统一观测 | Base Tools、Skill、MCP 的统一组装与包装 |

事件、审计、观测能力应尽量落到 Hook / Toolset Wrapper 层，而不是散落在业务接口里。

### Toolset 复用边界

当前阶段不需要把所有 Base Tools 一次性迁移到 FunctionToolset。更稳妥的方式是分阶段复用：

1. 短期保持 Base Tools 现有闭包函数列表，减少运行时扰动。
2. 中期优先引入 Toolset Wrapper，用于统一包装工具结果、事件推送和观测。
3. 对 MCP 工具优先使用框架原生 Toolset 能力，例如 FastMCPToolset、PrefixedToolset。
4. 等 deps 机制和子代理 clone 链路稳定后，再评估是否进一步 Toolset 化。

### MCP 接入架构决策

MCP 接入建议复用框架原生能力，并在其上增加轻量管理层和参数注入层：

- 工具发现和 schema 解析：复用 FastMCPToolset。
- 命名空间隔离：复用 PrefixedToolset。
- 连接生命周期、配置管理、刷新和容错：由 MCPClientManager 管理。
- 默认认证参数、租户参数、运行时上下文参数：由 DefaultArgsToolset 这类薄封装处理。

### Skill 系统设计决策

Skill 是面向特定任务的能力包，可能包含知识说明、执行脚本、工具调用策略和沙箱执行路径。执行层可以分为：

- 纯知识型 Skill：用于上下文注入和操作说明。
- Native Skill：可调用 base tools / MCP tools，适合低风险、轻量任务。
- Sandbox Skill：进入沙箱执行，适合代码生成、脚本执行、文件处理等高风险任务。
- Pi Agent Skill：适合需要较完整 agent loop 的复杂执行任务。

Skill 与 MCP 的边界需要清晰：MCP 偏外部服务工具接入，Skill 偏任务知识、执行策略和路径分流。

### 统一设计原则

- 框架原生能力优先复用，例如 `pydantic-deep`、`pydantic-ai`、`fastmcp`。
- 企业场景需要的管理层、适配层、策略层保留轻量封装。
- 配置优先，Provider、Model、Role、MCP 端点和默认参数都应配置化。
- 运行时策略集中收口，避免 provider 特判、安全策略、工具注入散落在业务层。
- 能力接入层并列管理 Base Tools、Skill、MCP、Sandbox。
- Hook、Capability、Toolset、Provider、Registry 各自职责清晰，减少重复包装和重复治理。

### 实施顺序

1. 先做多模型注册表收口，把模型名猜测和 provider 特判从业务层移出。
2. 再收口运行时策略，把 thinking、tool calling、streaming、compatibility 等差异统一处理。
3. 然后统一 Hook / Capability / Toolset 职责，用 Toolset Wrapper 承接工具包装、事件和观测。
4. 最后优化 Skill / MCP / Sandbox 的组合体验，形成稳定的能力接入层。

运行时架构的关键不是新增一个大模块，而是把执行内核、模型体系、行为治理和能力接入分层收口。后续新增模型、MCP 服务、Skill 或沙箱后端时，变更应落在明确的适配层和策略层，而不是扩散到业务代码里。

## 9. 风险清单与治理措施


主要风险包括：

- `pydantic-deep` 仍处于 pre-1.0 阶段。
- 多模型体系如果继续靠模型名猜测，会越来越难维护。
- 工具调用和沙箱执行存在安全风险。
- Skill 和 MCP 是外部能力入口，需要权限、隔离和审计。
- 静默异常、默认能力过宽、依赖私有 API 会影响生产稳定性。
- Agent 系统需要可观测、可回放和可追责能力。


#### 风险一：编排 SDK 版本稳定性

`pydantic-deep v0.3.3` 仍是 pre-1.0 版本，API 有变化风险。当前如采用它承担编排内核能力，必须通过工程手段控制升级风险。

治理措施：

- 精确锁定版本，例如 `pydantic-deep==0.3.3`。
- 升级前必须在 staging 环境验证。
- 对关键适配点增加集成测试。
- 保留降级到原生 `pydantic-ai` 的 fallback 路径。
- 必要时 fork 维护关键补丁。

#### 风险二：多模型体系不可持续

当前多模型体系如果继续通过模型名字符串判断 provider，例如通过模型名包含 `claude` 判断调用路径，会导致后续模型越来越难接入。

治理措施：

- 引入 model catalog。
- 建立 provider adapter。
- 建立 model registry。
- 建立 runtime policy。
- 把 streaming、tool calling、reasoning、thinking 等能力显式建模。
- 业务层只依赖逻辑角色，不直接感知 provider 差异。

#### 风险三：工具执行和沙箱安全

Agent 系统的高风险点不是模型回答，而是模型触发的工具调用、文件写入、脚本执行和网络访问。模型输出不能被视为授权。

治理措施：

- 工具调用必须经过风险识别、审批、执行、回写四个阶段。
- 默认最小权限。
- 高风险命令进入审计链路。
- 沙箱控制网络出口、文件系统范围和执行权限。
- 对 Skill、MCP、Shell、Filesystem 分别建立权限策略。
- 所有执行过程保留 trace 和审计日志。

#### 风险四：默认能力过宽

如果默认开启过多工具、Skill 或 MCP，容易增加误调用、token 浪费和安全风险。

治理措施：

- 默认能力收敛。
- 按执行模式注入工具。
- 按角色和上下文动态装配能力。
- 对高风险能力设置显式开关。

#### 风险五：异常不可见

Agent 系统链路长，如果静默吞异常，线上问题会很难定位。

治理措施：

- 禁止关键链路 `except Exception: pass`。
- 所有异常带上下文日志。
- 区分用户错误、工具错误、模型错误、系统错误和安全拒绝。
- 引入 tracing、metrics、事件流和错误码。

### 风险治理表

| 风险 | 影响 | 治理措施 |
|------|------|----------|
| SDK pre-1.0 | 升级断裂 | 锁版本、staging 验证、fallback、fork |
| 多模型硬编码 | 接入成本上升 | catalog、provider、registry、policy 分层 |
| 工具执行风险 | 文件、命令、网络风险 | 权限、审计、沙箱、默认拒绝 |
| 默认能力过宽 | 误调用和成本浪费 | 按模式动态注入，默认收敛 |
| 静默异常 | 线上不可排障 | 统一日志、错误分级、trace |
| Skill/MCP 外部入口 | 权限和供应链风险 | 白名单、隔离、审计、配置化 |

> 采用 `pydantic-deep` 不代表风险消失。我们的判断是：编排 SDK 风险可通过锁版本和 fallback 控制，真正需要重点投入的是安全治理、多模型治理、沙箱隔离、异常可观测和权限审计。这些能力需要结合企业场景沉淀在 Super-Agent 工程体系内。

## 10. 最佳工程实现内容

如果按 `07-工程最佳实践与安全设计汇总.md` 来提炼，最佳工程实现不是多加几个工具，而是把核心模块按一致的工程范式做完整。可以概括成下面这张表：

| 模块 | 最佳设计 | 反模式 | 工程结论 |
|------|----------|--------|----------|
| 核心循环设计 | 用 `AsyncGenerator` + 显式状态机统一多轮推理、工具调用和事件输出 | 用隐式回调拼主循环，状态散落在各处 | 主循环必须是可追踪、可中断、可回放的状态机 |
| 上下文管理 | `System Prompt`、运行时上下文、用户上下文三路拆分，静动态分离 | 把所有信息拼成一个大 prompt | 上下文需要分层、分优先级、可缓存 |
| Token 管理 | 做成多层降级压缩 Pipeline，优先低成本裁剪和后台预生成摘要 | 一超长就调模型做全量摘要 | 压缩系统要同时考虑成本、时延和连续性 |
| 长期记忆 | 采用克制写入策略，后台提取 + 周期整合 + 垃圾回收 | 只追求“记得更多”，最后变成噪声库 | 记忆系统核心是治理，不是堆积 |
| 工具系统 | 统一工具接口，调用前后做过滤、包装、审计和结果回写 | 每个工具各写一套协议和返回格式 | 工具层要先统一接口，再谈扩展 |
| 工具能力层 | 基础工具、MCP、Skills 按能力层并列接入，而不是互相耦合 | 把所有能力混成单一工具包 | 能力接入层要标准化、组件化 |
| Agent 工具生态 | 工具设计和权限模型一起考虑，执行前先分级和隔离 | 把工具调用等同于普通函数调用 | 工具生态设计必须默认带安全边界 |
| 从传统 API 到 Agent Tool | 用标准 SOP 把 API 转成 Tool：输入、输出、权限、错误、审计一次定义清楚 | 直接把现有 API 暴露给模型 | Tool 化必须是工程转换，不是简单包装 |
| MCP 接入 | 复用协议生态，增加连接管理、命名空间和默认参数注入 | 自己重写协议，或直接裸用外部 MCP | MCP 要“生态复用 + 企业增强” |
| 执行能力层 | 文件、代码解释器、浏览器三类执行能力独立封装并隔离 | 让模型直接碰宿主机文件、命令和网络 | 执行能力必须通过隔离环境统一治理 |
| Skills | 把知识说明、执行脚本、工具策略、沙箱路径封装成能力包 | 把 Skill 简化成 prompt 模板或 bash 脚本 | Skill 应该是知识与执行的组合抽象 |
| 记忆工程总论 | 把记忆当成系统级能力，定义写入标准、读取策略和治理边界 | 只讨论“记不记”，不讨论“怎么治理” | 记忆工程要有完整生命周期视角 |
| 上下文生命周期管理 | 做好 Bootstrap、剪枝、压缩、恢复全链路 | 只管往上下文里堆内容，不做生命周期控制 | 上下文是运行时资源，需要全过程管理 |
| 知识记忆写入与治理 | 建立写入、修订、GC 闭环，保证知识库长期可用 | 只会新增，不会清理和纠偏 | 知识记忆必须具备治理和回收机制 |

### 简要总结

如果把这 14 个模块再压缩一层，最佳工程实现实际上是在做四件事：

1. 把主循环、上下文、压缩、记忆做成稳定运行时，而不是 prompt 技巧堆叠。
2. 把工具、MCP、Skill、执行能力做成统一能力接入层，而不是零散插件集合。
3. 把权限、安全、审计、隔离做成默认内建能力，而不是上线前补丁。
4. 把知识、状态、观测、治理做成闭环，保证系统长期可维护。

这也说明这个项目的重点不在“调用哪个框架”，而在把这些模块按工程标准做完整。

## 11. 建设范围


从现有模块规划看，下一代企业智能体已经是一个中型平台工程。它覆盖了从入口网关、流式协议、状态管理、模型接入、智能体编排、沙箱执行、Skill、MCP、安全、可观测、计费、前端交互到部署运维的完整链路。

其中技术重心主要在运行时控制、安全治理、模型适配和工具执行闭环，而不是单纯 UI 展示。

前端也不是简单聊天框。它需要支持 SSE/WebSocket 流式响应、A2UI 动态组件、工具结果展示、Sub-Agent 状态展示、终端输出、产物预览、模型切换和 Skill 交互。

### 模块概览

| 模块 | 目标 |
|------|------|

| API 网关与会话入口 | 请求入口、会话、取消、管理接口 |
| 流式协议与事件通道 | SSE/WebSocket、事件协议、恢复机制 |

| 基础工具层与 Worker 体系 | 工具抽象、Worker 协议、事件发布 |

| 技能系统 | Skill 发现、加载、执行、创建 |
| MCP 工具接入 | MCP 配置、连接、刷新、容错 |
| 记忆系统 | 用户事实、检索、更新、摘要压缩 |

这部分范围覆盖的是完整产品能力，不是单一 Demo 或单一 Agent 链路。

建议不要一次性追求完整平台，而是分阶段交付。先证明主链路可运行，再补齐执行闭环，之后做企业化安全、观测和产品体验打磨。

## 13. 需要领导决策的事项


建议领导重点决策以下事项：

- 是否确认 `pydantic-deep` 作为当前阶段 Agent 编排内核。
- 是否确认“轻量编排内核 + 工程化控制面 + 安全治理加固”的总体路线。
- 是否认可 Pi-mono 继续作为沙箱执行主方案。
- 是否认可运行时组件按“模型注册表、运行时策略、Hook/Capability/Toolset、Skill/MCP/Sandbox 能力接入层”分阶段收口。
- 是否确认完整建设范围边界。
- 是否同意按四阶段推进。
- 是否优先投入安全、沙箱、模型路由、观测等企业化能力。


本次技术调研已经形成比较明确的工程判断，但后续推进仍需要在方向、资源和范围上做决策。

第一，需要确认 Agent 编排 SDK 方向。如果确认采用 `pydantic-deep` 承担编排内核能力，团队后续可以围绕它锁版本、补适配、做 fallback 和集成测试。如果方向反复，后续主控编排、Sub-Agent、Toolset、Skill、MCP 都会受到影响。

第二，需要确认工程化沉淀与复用边界。建议通用能力优先复用，平台控制面结合企业场景自研沉淀，这是当前较平衡的路线。

第三，需要确认沙箱执行主链路。Pi-mono 当前综合性价比较高，Claude API 直调作为补充，Claude Code CLI 作为能力参考。这个决策会影响 Worker、Sandbox Manager、Token Manager 和安全策略设计。

第四，需要确认建设范围边界。如果只做早期原型或试点验证，可以裁剪范围；如果要企业化交付，则安全、观测、测试和部署不能省。

第五，需要确认阶段目标。建议先做最小闭环，再逐步增强执行闭环、企业化能力和产品体验，避免项目一开始范围过大导致交付周期失控。

### 建议决策清单

| 决策事项 | 建议结论 | 原因 |
|----------|----------|------|
| Agent 编排 SDK | 采用 `pydantic-deep v0.3.3` | 架构匹配、依赖轻、可降级 |
| 总体路线 | 轻量编排内核 + 工程化控制面 | 平衡复用效率和长期自主权 |
| 沙箱执行主方案 | 继续使用 Pi-mono | 功能完整，综合成本低 |
| 运行时组件治理 | 分阶段收口模型、策略、工具和能力接入层 | 降低运行时逻辑分散和维护成本 |
| Claude API 直调 | 作为轻量补充 | 适合简单场景 |
| Claude Code CLI | 作为能力参考 | 上下文和成本开销较高 |
| 建设范围 | 完整产品能力边界 | 覆盖可交付、可联调、可测试、可演示 |
| 推进方式 | 四阶段推进 | 降低一次性交付风险 |

### 最终汇报结论

> 第一，Super-Agent 当前阶段建议采用 `pydantic-deep` 承担 Agent 编排内核能力，它不是能力最全的框架，但与当前架构契合度较高。
>
> 第二，项目后续重点不是比较框架优劣，而是基于轻量编排内核补齐工程化控制面，包括多模型注册表、运行时策略、Hook/Capability/Toolset 分层治理、Skill/MCP/Sandbox 能力接入、安全权限、事件流和可观测。
>
> 第三，建议先确认完整建设范围，再按四个阶段推进，先跑通主链路，再补齐执行闭环和企业化能力。

## 附录：

本次调研的结论可以压缩为四点。

第一，编排内核建议采用 `pydantic-deep v0.3.3`，因为它轻量、可嵌入、可演进，和当前的主控编排分层设计契合度较高。

第二，其他框架更适合作为专项能力参考：DeerFlow 参考中间件和安全审计，AgentScope 参考多 Agent、记忆和观测，Deep Agents 参考执行抽象和权限模型，CrewAI 参考固定流程编排场景。

第三，沙箱执行层建议继续使用 Pi-mono；运行时架构需要把多模型、策略、工具、Skill、MCP 和安全能力分层收口，避免逻辑散落在业务代码里。

第四，后续建设路线建议保持“轻量编排内核 + 工程化控制面 + 安全治理加固”，并按四个阶段逐步推进。
