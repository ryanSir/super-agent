# Super-Agent 技术调研

> 汇报主题：Super-Agent 前期技术调研、框架选型、自研边界与实施计划  
> 推荐受众：公司领导、业务部门负责人、技术负责人、项目决策人  
> 汇报目标：从技术角度说明下一代企业智能体的框架调研结论、共性能力价值、业务团队接入方式、工程风险和阶段性建设计划。
> 调研边界：本文只讨论下一代企业智能体的技术实现路径和工程组合方式，不涉及对既有产品线、既有 Agent 能力或其他团队方案的业务评价。

---

## 1. 调研背景与目标

我们的目标是构建下一代企业智能体，面向企业内部复杂知识工作、工具协同和业务执行场景，形成可持续演进、可维护、可治理的智能体能力体系。

核心判断维度包括：Agent 编排能力如何复用成熟技术，模型、工具、Skill、MCP、沙箱、安全审计和可观测能力如何形成统一标准，以及哪些控制面能力需要结合企业场景做工程化建设。

整体思路是：成熟能力优先复用，关键控制面按企业场景沉淀，避免重复建设，也避免过度耦合。

### 现有产品线技术架构背景

从前期背景调研看，当前各产品线已经在不同业务场景中形成了各自的 Agent 技术实现路径。

| 产品线 / 场景 | 当前技术架构 | 技术特征 |
|---------------|--------------|----------|
| R&D | OpenAI SDK,手动实现 agent loop| 类型体系清晰，适合在 Python 工程内做轻量 Agent 能力沉淀 |
| ip-drafting | `LangGraph` + `LangChain`，手动实现 agent loop | 图编排能力强，适合流程明确、状态流转复杂的任务 |
| ls | `CrewAI` | 角色和任务抽象清晰，适合固定流程、角色边界明确的业务 |
| analytics | `Claude Code SDK` | 工具链完整，适合代码、分析和项目上下文较重的场景 |
| mse | `LangChain` + `LangGraph` 手动实现 agent loop | 生态完整，适合基于 LangChain 体系构建复杂链路 |
| ip-checking | `LangGraph` + OpenAI SDK，手动实现 agent loop | 灵活度高，适合对执行循环和模型调用有定制要求的场景 |

企业内部 Agent 技术路线已经呈现多框架、多实现、多运行时并存的状态。不同产品线根据自身业务诉求做了合理取舍，但从下一代企业智能体的技术建设角度看，也带来几个共性问题：

- Agent 编排层实现方式不统一，后续能力复用和跨场景迁移成本较高。
- 多模型接入、工具调用、MCP、Skill、沙箱、安全审计等能力容易在不同产品线重复建设。
- 手动实现 agent loop 的方案灵活度高，但长期需要承担上下文管理、工具执行、错误恢复、观测和安全治理成本。
- LangChain / LangGraph / CrewAI / Claude Code SDK 等方案各有优势，但运行时假设、依赖体系和扩展方式不同，需要在企业级平台层做统一抽象。

因此，本次 Super-Agent 技术路线会在充分理解现有技术分布的基础上，沉淀出一套更适合长期维护、统一治理和跨场景复用的企业智能体工程架构。

从这个背景出发，后续技术方案需要突出几个特点：

- 编排内核要轻量，避免把业务系统过度绑定到单一大型框架。
- 多模型、工具、Skill、MCP、沙箱和安全策略要有统一治理层。
- 运行时能力要可观测、可审计、可回放，满足企业级维护要求。
- 通用能力优先复用，差异化控制面在 Super-Agent 工程体系内沉淀。

## 2. 候选方案范围

本次调研覆盖三类对象：

- SDK 型框架：`pydantic-deep-agent`(pydantic-ai)、Deep Agents(langchain)、CrewAI。
- 应用型或平台型框架：DeerFlow、AgentScope、Hermes。
- 沙箱执行引擎：Pi-mono CLI、Claude Code CLI、Claude API 直调。

SDK 型框架更适合作为编排组件集成到工程体系中；应用型或平台型框架更适合作为工程设计、平台能力和安全治理参考；沙箱执行引擎单独评估，它直接影响代码、脚本、Skill 和工具执行的安全边界。

### 框架定位对比

| 方案 | 类型 | 技术特长 | 关注点 | 在 Super-Agent 中的定位 |
|------|------|--------|----------|--------------------------|
| pydantic-deep | SDK 型 | 轻量、可嵌入、依赖少，适合嵌到现有系统里做底层编排 | pre-1.0，部分能力需补强 | 作为当前编排内核候选 |
| Deep Agents | SDK 型 | 对执行环境隔离、权限控制和安全风险防范考虑较完整 | API 波动大，LangChain 依赖重 | 作为执行抽象参考 |
| CrewAI | SDK 型 | 流程和角色分工表达直接，适合把任务拆成固定步骤来跑 | 更偏固定流程编排，角色调优成本较高 | 固定流程场景参考 |
| DeerFlow | 应用型 | 完整工程实现比较成熟，尤其是流程拦截、安全审计和异常处理 | LangChain/LangGraph 锁定重 | 工程质量参考 |
| AgentScope | 平台型 | 多 Agent 协作、记忆管理、状态维护和运行观测能力较完整 | 无内置沙箱，核心实现偏重 | 多 Agent 能力蓝图 |
| Hermes | 应用型 | 多入口接入和工具连接能力较强，偏面向具体业务场景集成 | 自研逻辑重，通用 SDK 属性弱 | 局部能力参考 |

## 3. 选型判断标准

选型重点从“框架功能完整度”转向“工程适配度”。Super-Agent 已经具备请求入口、模型路由、主控编排层、上下文装配、工具注入、子任务协作、事件流输出和安全治理等控制面，因此 Agent SDK 更适合作为编排能力组件，而不是主导整个系统结构。

核心判断标准如下：

### 判断标准表

| 判断维度 | 关注问题 | 对 Super-Agent 的重要性 |
|----------|----------|--------------------------|
| 架构契合度 | 是否匹配动态路由、工具调用和子任务协作 | 最高 |
| 依赖风险 | 是否深度绑定 LangChain 等重依赖 | 高 |
| 可演进性 | 是否能降级、扩展或 fork | 高 |
| 工程可控性 | 团队能否自行修复和维护 | 高 |
| 安全治理 | 权限、审计、沙箱是否可控 | 最高 |
| 扩展能力 | 是否支持后续平台化演进 | 中高 |

## 4. 技术路线建议：采用 pydantic-deep 作为编排内核

工程组合建议以 `pydantic-deep v0.3.3` 承担 Agent 编排内核能力。它的核心价值在于轻量、低耦合，以及与 `pydantic-ai` 生态的天然衔接。

需要说明的是，`pydantic-deep` 并不是重写了一套底层模型调用框架，它底层仍然基于 `pydantic-ai`。`pydantic-ai` 主要负责模型调用、Provider 适配、tool calling、消息结构和基础 Agent 循环；`pydantic-deep` 则在这个基础上补了一层更适合 Agent 工程化落地的运行时能力，例如 Agent 工厂化组装、子任务委派、Toolset 体系、Hooks / Capabilities、上下文管理、Skills、Checkpoint、Memory 和成本追踪等。

`pydantic-deep` 在保留 `pydantic-ai` 底层能力的前提下，提前封装了复杂 Agent 运行时所需要的一批工程能力，可以减少我们在编排内核层的重复建设。


### 核心优势

| 优势 | 说明 |
|------|------|
| 编排模型契合 | LLM 自主决策 + 子任务委派，与主控编排分层天然匹配 |
| 依赖链轻 | 主要依赖 `pydantic-ai`，避免深度绑定 LangChain/LangGraph |
| 可降级 | 必要时可以退回原生 `pydantic-ai` |
| 可维护 | 源码规模较小，团队可阅读、patch 或 fork |
| 集成成本低 | 对现有架构侵入较小，可以与当前控制面协同 |


> 当前建议以 `pydantic-deep` 承担 Agent 编排内核能力。主要考虑是它轻、可嵌入、依赖少、可降级，和我们当前的主控编排、子任务协作、工具注入和动态路由设计契合度较高。

## 5. 面向业务部门的接入方式与最佳实践

如果 ls、mse、analytics、ip-drafting、ip-checking 等业务团队关注“如何对接或使用 pydantic 体系”，建议先区分两个层次：

- `pydantic-ai`：偏底层 Agent SDK，解决模型接入、工具调用、结构化输出、消息历史和基础 Agent 循环。
- `pydantic-deep`：偏 Agent 运行时增强层，在 `pydantic-ai` 之上补充上下文管理、Hooks / Capabilities、Toolset、Skills、子任务委派、Checkpoint、Memory、成本追踪等工程能力。

对业务部门的直接价值主要有四点：

- 降低重复建设：模型配置、工具接入、MCP、沙箱、安全审计、事件流和观测不需要各团队重复实现。
- 保留现有积累：已有 LangGraph、CrewAI、Claude Code SDK 或自研 Agent 可以保留主流程，先对齐共性能力。
- 提升企业化交付能力：高风险执行进入沙箱，关键工具调用有审计，运行过程可观测、可排障。
- 降低后续协作成本：不同业务能力可以通过 Tool、MCP、Skill 或 Worker 进入统一能力体系，便于跨场景复用。

业务团队不需要一开始就完整接入所有能力，可以按场景选择接入层级：

| 场景 | 建议方式 | 说明 |
|------|----------|------|
| 只需要模型调用、结构化输出、简单工具调用 | 直接使用 `pydantic-ai` | 适合轻量智能化能力，接入成本最低 |
| 需要 Agent 自主调用工具、上下文管理、生命周期扩展 | 使用 `pydantic-deep` | 适合中复杂度 Agent 场景 |
| 已有 LangGraph、CrewAI、Claude Code SDK 或自研 Agent 实现 | 保留现有主流程，按需对齐通用能力 | 可优先对齐模型治理、工具协议、事件流、安全审计和沙箱策略 |
| 需要接入 Super-Agent 能力体系 | 以 Tool / MCP / Skill / Worker 形式接入 | 业务能力作为可调用能力进入统一编排和治理链路 |

推荐的最佳实现方式是：

1. 模型不要散落在业务代码里写死，优先通过模型目录、角色绑定和运行时策略统一管理。
2. 业务能力不要直接暴露给模型执行，优先包装成 Tool、MCP、Skill 或 Worker，并经过权限、审计和观测。
3. 高风险能力，例如代码执行、文件操作、脚本运行，应进入沙箱执行层。
4. 前端交互不要只依赖最终文本，建议通过事件流表达思考、步骤、工具结果、产物和异常状态。
5. 各业务团队可以按能力逐步接入，不需要一次性改造完整 Agent 架构。

## 6. 其他框架的技术定位与借鉴价值


- DeerFlow：工程质量、安全审计和中间件设计成熟，适合作为工程治理参考。
- AgentScope：多 Agent、记忆系统和观测能力完整，适合作为平台能力蓝图参考。
- Deep Agents：执行抽象、权限模型和威胁建模清晰，适合作为沙箱与权限设计参考。
- CrewAI：显式流程、角色化编排和商业化成熟度较高，适合固定流程任务参考。
- Hermes：多通道和工具接入有局部亮点，适合作为工具生态接入参考。


#### DeerFlow

DeerFlow 的优势非常明显，尤其是在中间件体系、安全审计和错误处理上。它的有序中间件、工具错误处理、循环检测和沙箱审计都值得 Super-Agent 参考。

从集成角度看，DeerFlow 重度依赖 LangChain/LangGraph，整体更像一个完整 Agent 应用，而不是轻量 SDK。如果作为组件集成，需要评估它的图结构、运行时假设和工程组织方式带来的维护成本。

因此 DeerFlow 的定位更适合作为工程质量和安全设计参考。

#### AgentScope

AgentScope 在多 Agent 协作、状态管理、记忆系统和可观测性方面非常成熟。它的 MsgHub、状态序列化、分层记忆和 OTel 观测都具有参考价值。

从集成角度看，它没有内置沙箱能力，命令执行和安全审计需要额外补齐。同时它的核心实现相对偏重，需要结合现有控制面边界谨慎拆解。

因此 AgentScope 更适合作为未来多 Agent 平台能力蓝图。

#### Deep Agents

Deep Agents 的优点是抽象边界清晰，尤其是 BackendProtocol、PermissionsMiddleware、SummarizationMiddleware 和威胁建模意识。这些能力对 Super-Agent 很有价值。

从工程落地角度看，它的 API 变化较快，版本稳定性需要观察，同时依赖 LangChain 生态，升级风险和长期维护成本需要评估。

因此 Deep Agents 更适合作为执行后端抽象和权限模型的设计参考。

#### CrewAI

CrewAI 的优势是流程化、角色化和商业化成熟，适合固定流程任务。例如固定研究流程、固定内容生产流程或固定审批流程。

Super-Agent 的目标是动态路由和动态委派。我们希望系统根据请求实时判断是否需要规划、是否需要工具、是否需要子代理。CrewAI 的显式编排方式更适合流程稳定、角色明确的任务场景。

此外，前期使用中已经发现 CrewAI 的角色、backstory、manager 行为需要较多调优，集成成本偏高。

#### Hermes

Hermes 在多通道接入和工具集成上有一些局部亮点，整体工程假设较强。对于当前 Super-Agent 来说，它更适合作为局部能力参考。

### 对比结论

| 框架 | 技术关注点 | 可借鉴价值 |
|------|------------------|----------|
| DeerFlow | 应用型框架，LangChain/LangGraph 依赖较重 | 安全审计和中间件 |
| AgentScope | 平台能力完整，沙箱能力需额外补齐 | 多 Agent、记忆和观测 |
| Deep Agents | API 变化较快，LangChain 依赖较重 | 执行抽象和权限模型 |
| CrewAI | 显式编排，更适合固定流程 | 固定流程任务参考 |
| Hermes | 工程假设较强，通用 SDK 属性较弱 | 工具和多通道接入参考 |

> 其他框架都有各自的技术价值。本次调研的结论是把不同框架的优势放到合适的位置：编排内核采用轻量可控的 `pydantic-deep`，工程治理参考 DeerFlow，多 Agent 和记忆参考 AgentScope，执行抽象和权限模型参考 Deep Agents。

## 7. 沙箱执行引擎选型

沙箱执行引擎单独评估，不直接跟随 Agent SDK。Super-Agent 的高风险执行场景包括代码生成、文件操作、脚本执行、Skill 执行、工具调用和结果文件回收，这些能力必须通过沙箱执行引擎隔离。

综合功能完整度、成本、工程投入和现有集成情况，当前建议继续以 Pi-mono 作为沙箱主链路；Claude API 直调作为轻量补充；Claude Code CLI 保留为能力参考。

### 测试数据摘要

基于 12 个典型场景做了对比测试，覆盖简单任务、中等任务、复杂任务、Skill 执行和端到端任务。
| 维度 | Pi-mono | Claude API 直调 | Claude Code CLI |
|------|---------|----------------|----------------|
| 简单任务（case 1-3） | 23.9s | 22.0s | 40.8s |
| 中等任务（case 4-6） | 47.0s | 49.8s | 64.4s |
| 复杂任务（case 7-8） | 61.7s | 69.6s | 77.3s |
| Skill 执行（case 10-11） | 41.3s | 34.0s | 50.3s |
| 端到端（case 12） | 24.4s | 26.1s | 38.0s |
| 总耗时（11 case） | **198.3s** | **201.5s** | **270.8s** |
| 胜场 | 6 | 5 | 0 |
| 成功率（11 case） | 100% | 100% | 100% |
| 输入 token 开销/次 | 不透明 | ~900-4,000 | ~74,000-308,000 |
| 总成本 | 不透明（走 gateway） | 低 | ~$8.89（11 次调用） |

测试数据说明：

- Pi-mono 与 Claude API 直调整体性能非常接近，11 个有效 case 总耗时仅差 3.2s。
- Pi-mono 在多步和复杂任务上更快，中等任务和复杂任务两组都领先。
- Claude API 直调在读取密集型任务和部分 Skill 执行上更快。
- Claude Code CLI 功能完整，但速度最慢，且 token 和成本开销显著更高。
- Case 9（REST API 服务器）中，Pi-mono 出现 180s 超时，Claude API 为 42.7s，Claude Code CLI 为 28.9s；该 case 未计入上表汇总统计。

### 核心对比

| 方案 | 优点 | 问题 | 结论 |
|------|------|------|------|
| Pi-mono CLI | 完整 agent loop、并行工具、session、hooks、compaction、多模型 | Node.js 子进程启动有一定开销，JSONL 解析需适配 | 继续作为主方案 |
| Claude API 直调 | 进程内调用，最小上下文，轻量可控 | 需要自建 loop、streaming、并行工具、hooks、session、压缩 | 作为补充方案 |
| Claude Code CLI | 项目感知和工具链完整 | 上下文开销较高，成本和时延需要控制 | 保留为能力参考 |

测试结论是：Pi-mono 与 Claude API 直调整体性能接近，但 Pi-mono 已经具备完整 agent loop、并行工具、上下文压缩、session、hooks 和多模型协议支持。如果用 Claude API 直调补齐这些能力，需要额外建设周期。Claude Code CLI 功能完整，但上下文和成本开销较高。

## 8. 自研与复用边界

总体原则是：通用能力优先复用，平台控制面做工程化建设。Agent loop、模型调用、基础工具包装、MCP 连接等能力已有成熟生态；网关、状态、编排、能力接入、安全治理、观测和前端交互能力则需要结合企业场景统一设计。

### 边界划分

| 能力模块 | 建议方式 | 原因 |
|----------|----------|------|
| 基础工程与配置底座 | 工程化沉淀 | 需要统一项目结构、配置管理、日志规范、构建和部署约束 |
| API 网关与会话入口 | 工程化沉淀 | 需要统一请求入口、会话管理、取消控制和管理接口 |
| 流式协议与事件通道 | 工程化沉淀 | 需要统一 SSE / WebSocket / A2UI 事件协议和恢复机制 |
| 会话状态与上下文持久化 | 工程化沉淀 | 需要管理 Session 生命周期、历史裁剪、TTL 和恢复 |
| Agent loop | 复用 `pydantic-deep` | 降低自研成本，保持轻量嵌入 |
| 模型基础调用 | 复用 `pydantic-ai` | 生态稳定，类型友好 |
| Hooks / Capabilities | 复用 `pydantic-deep` + 工程化扩展 | 框架已提供生命周期钩子和行为注入机制，上层补审计、事件发布和策略控制 |
| 上下文管理 / 历史压缩 | 复用 `pydantic-deep` + 工程化策略 | 框架已有上下文管理能力，上层需要补阈值策略、归档和恢复机制 |
| Prompt / Context 装配 | 工程化沉淀 | 需要按模式、角色、用户上下文和记忆统一装配输入上下文 |
| 主控编排 / 调度层 | 工程化沉淀 | 关系到业务模式路由和产品差异化 |
| MCP 基础连接 | 复用 `fastmcp` / `pydantic-ai` | 避免重复建设协议层 |
| Worker / Toolset 组织层 | 复用 `pydantic-deep` Toolset + 工程化组织 | 框架提供工具组织能力，上层仍需统一工具包装、结果转换、事件推送和能力编排 |
| 沙箱内执行 | 复用 Pi-mono | 功能完整，成本可控 |
| 多模型路由 | 自研薄层 | 需要适配内部模型、网关和角色策略 |
| 子任务 / 多代理协作 | 复用 `pydantic-deep` + 工程化约束 | 框架已有子任务委派能力，上层仍需管理并发限制、生命周期和结果聚合 |
| Skill 系统 | 复用框架能力 + 工程化扩展 | 框架可提供技能运行时，业务侧仍需补知识模板、脚本执行治理和沙箱分流 |
| 记忆系统 | 工程化沉淀（当前以自建为主） | 用户事实、会话压缩、检索增强和历史复用更贴近企业业务约束 |
| Checkpoint / 会话快照 | 按需复用 `pydantic-deep` | 框架已有基础能力，但是否启用取决于会话恢复和回滚场景需要 |
| 安全权限 | 自研或强控制 | 不能完全交给模型或第三方框架 |
| 事件流协议 | 工程化沉淀 | 直接影响前端 A2UI 和用户体验 |
| 可观测与计费 | 自研集成 | 关系到企业化交付和运营 |
| 前端 A2UI 渲染与业务组件 | 工程化沉淀 | 需要把思考、步骤、工具结果、子代理状态和产物统一可视化 |
| 测试体系与质量保障 | 工程化沉淀 | 需要覆盖接口、流式链路、Agent / Worker 集成和回归测试 |
| 部署、运维与国际化 | 工程化沉淀 | 关系到环境落地、排障能力和多语言交付 |

这里的关键边界不是“框架能不能做”，而是“哪些能力可以复用底层运行时，哪些能力必须在企业工程体系里统一收口”。模型输出不是授权，所有工具执行都要经过系统策略层；权限、安全、审计和沙箱策略必须由系统独立校验。

## 9. 最佳工程实现内容

如果按 `07-工程最佳实践与安全设计汇总.md` 来提炼，最佳工程实现不是多加几个工具，而是把核心模块按一致的工程范式做完整。可以概括成下面这张表：

| 模块 | 最佳设计 | 反模式 | 工程结论 |
|------|----------|--------|----------|
| 核心循环设计 | 用 `AsyncGenerator` + 显式状态机统一多轮推理、工具调用和事件输出 | 用隐式回调拼主循环，状态散落在各处 | 主循环必须是可追踪、可中断、可回放的状态机 |
| 上下文管理 | `System Prompt`、运行时上下文、用户上下文三路拆分，静动态分离 | 把所有信息拼成一个大 prompt | 上下文需要分层、分优先级、可缓存 |
| Token 管理 | 做成多层降级压缩 Pipeline，优先低成本裁剪和后台预生成摘要 | 一超长就调模型做全量摘要 | 压缩系统要同时考虑成本、时延和连续性 |
| 长期记忆 | 采用克制写入策略，后台提取 + 周期整合 + 垃圾回收 | 只追求“记得更多”，最后变成噪声库 | 记忆系统核心是治理，不是堆积 |
| 工具系统 | 统一工具接口，调用前后做过滤、包装、审计和结果回写 | 每个工具各写一套协议和返回格式 | 工具层要先统一接口，再谈扩展 |
| 工具能力层 | 基础工具、MCP、Skills 按能力层并列接入，而不是互相耦合 | 把所有能力混成单一工具包 | 能力接入层要标准化、组件化 |
| Agent 工具生态 | 工具设计和权限模型一起考虑，执行前先分级和隔离 | 把工具调用等同于普通函数调用 | 工具生态设计必须默认带安全边界 |
| 从传统 API 到 Agent Tool | 用标准 SOP 把 API 转成 Tool：输入、输出、权限、错误、审计一次定义清楚 | 直接把现有 API 暴露给模型 | Tool 化必须是工程转换，不是简单包装 |
| MCP 接入 | 复用协议生态，增加连接管理、命名空间和默认参数注入 | 自己重写协议，或直接裸用外部 MCP | MCP 要“生态复用 + 企业增强” |
| 执行能力层 | 文件、代码解释器、浏览器三类执行能力独立封装并隔离 | 让模型直接碰宿主机文件、命令和网络 | 执行能力必须通过隔离环境统一治理 |
| Skills | 把知识说明、执行脚本、工具策略、沙箱路径封装成能力包 | 把 Skill 简化成 prompt 模板或 bash 脚本 | Skill 应该是知识与执行的组合抽象 |
| 记忆工程总论 | 把记忆当成系统级能力，定义写入标准、读取策略和治理边界 | 只讨论“记不记”，不讨论“怎么治理” | 记忆工程要有完整生命周期视角 |
| 上下文生命周期管理 | 做好 Bootstrap、剪枝、压缩、恢复全链路 | 只管往上下文里堆内容，不做生命周期控制 | 上下文是运行时资源，需要全过程管理 |
| 知识记忆写入与治理 | 建立写入、修订、GC 闭环，保证知识库长期可用 | 只会新增，不会清理和纠偏 | 知识记忆必须具备治理和回收机制 |

### 简要总结

如果把这 14 个模块再压缩一层，最佳工程实现实际上是在做四件事：

1. 把主循环、上下文、压缩、记忆做成稳定运行时，而不是 prompt 技巧堆叠。
2. 把工具、MCP、Skill、执行能力做成统一能力接入层，而不是零散插件集合。
3. 把权限、安全、审计、隔离做成默认内建能力，而不是上线前补丁。
4. 把知识、状态、观测、治理做成闭环，保证系统长期可维护。

这也说明这个项目的重点不在“调用哪个框架”，而在把这些模块按工程标准做完整。

## 10. 建设范围

## 11. 阶段性推进计划


建议按四个阶段推进，先验证主链路，再补齐执行闭环、安全治理和产品体验。

### 阶段计划表

| 阶段 | 核心目标 | 交付结果 |

## 12. 需要领导决策的事项


建议领导重点决策以下事项：

- 是否确认 `pydantic-deep` 作为当前阶段 Agent 编排内核。
- 是否确认“轻量编排内核 + 工程化控制面 + 安全治理加固”的总体路线。
- 是否认可 Pi-mono 继续作为沙箱执行层的主要执行引擎。
- 是否认可模型、工具、Skill、MCP、沙箱和事件流等工程能力分阶段收口。
- 是否确认完整建设范围边界。
- 是否同意按四阶段推进。
- 是否优先投入安全、沙箱、模型路由、观测等企业化能力。


本次技术调研已经形成比较明确的工程判断，但后续推进仍需要在方向、资源和范围上做决策。

第一，需要确认 Agent 编排 SDK 方向。如果确认采用 `pydantic-deep` 承担编排内核能力，团队后续可以围绕它锁版本、补适配、做 fallback 和集成测试。如果方向反复，后续主控编排、子任务协作、Toolset、Skill、MCP 都会受到影响。

第二，需要确认工程化沉淀与复用边界。建议通用能力优先复用，平台控制面结合企业场景自研沉淀，这是当前较平衡的路线。

第三，需要确认沙箱执行引擎。Pi-mono 当前综合性价比较高，Claude API 直调作为补充，Claude Code CLI 作为能力参考。这个决策会影响 Worker、Sandbox Manager、Token Manager 和安全策略设计。

第四，需要确认建设范围边界。如果只做早期原型或试点验证，可以裁剪范围；如果要企业化交付，则安全、观测、测试和部署不能省。

第五，需要确认阶段目标。建议先做最小闭环，再逐步增强执行闭环、企业化能力和产品体验，避免项目一开始范围过大导致交付周期失控。

### 建议决策清单

| 决策事项 | 建议结论 | 原因 |
|----------|----------|------|
| Agent 编排 SDK | 采用 `pydantic-deep v0.3.3` | 架构匹配、依赖轻、可降级 |
| 总体路线 | 轻量编排内核 + 工程化控制面 | 平衡复用效率和长期自主权 |
| 沙箱执行引擎 | 继续使用 Pi-mono | 功能完整，综合成本低 |
| 工程能力治理 | 分阶段收口模型、工具、Skill、MCP、沙箱和事件流 | 降低运行逻辑分散和维护成本 |
| Claude API 直调 | 作为轻量补充 | 适合简单场景 |
| Claude Code CLI | 作为能力参考 | 上下文和成本开销较高 |
| 建设范围 | 完整产品能力边界 | 覆盖可交付、可联调、可测试、可演示 |
| 推进方式 | 四阶段推进 | 降低一次性交付风险 |

## 13. 最终总结

1. 总体技术路线：下一代企业智能体建设不应依赖单一“大而全”框架完成所有能力，而应采用“轻量编排内核 + 工程化控制面 + 安全治理加固”的技术路线。底层编排能力优先复用成熟生态，企业场景相关的模型治理、能力接入、安全权限、事件流、观测和交付能力在工程体系内统一收口。

2. 编排内核建议：建议采用 `pydantic-deep v0.3.3`。它底层基于 `pydantic-ai`，能够复用模型、工具调用和 Agent 基础能力，同时在上层补充 Agent 工程化运行时所需的上下文管理、Hooks / Capabilities、Toolset、Skills、子任务委派等能力。它不是功能最重的框架，但更适合作为当前阶段可嵌入、可维护、可降级的编排内核。

3. 其他框架定位：其他框架不作为互斥选择处理，而是作为专项能力参考。DeerFlow 适合参考中间件、安全审计和错误处理；AgentScope 适合参考多 Agent、记忆和状态管理；Deep Agents 适合参考执行环境、权限控制和安全风险治理；CrewAI 适合参考固定流程和角色分工明确的编排场景。

4. 沙箱执行建议：沙箱执行层建议继续以 Pi-mono 作为主要执行引擎。测试数据表明，Pi-mono 与 Claude API 直调整体性能接近，但 Pi-mono 已经具备完整 agent loop、并行工具、上下文压缩、session、hooks 和多模型协议支持。Claude API 直调可以作为轻量补充，Claude Code CLI 更适合作为能力参考。

5. 业务团队价值：这套方案的价值不是要求各团队统一到同一套实现，而是提供可复用的工程能力边界。轻量场景可以直接使用 `pydantic-ai`，复杂 Agent 场景可以使用 `pydantic-deep`，已有系统可以优先对齐模型治理、工具接入、安全审计、沙箱执行、事件流和可观测等共性能力。

6. 后续推进建议：建议按阶段交付，先跑通最小可运行闭环，再补齐工具、Skill、沙箱和协作执行能力，之后强化企业化安全、观测、测试和部署，最后做产品化体验和稳定性打磨。这样可以在控制风险的前提下，逐步形成可持续演进、可维护、可治理的企业智能体能力体系。
